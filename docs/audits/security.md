# Security audit – agora

Date: 2026-04-17
Scope: `apps/api`, `apps/web`, `docker-compose.yml`, `apps/web/nginx.conf`.
Mode: read-only, static review. No tests or containers were run.

## Executive summary

The core primitives are competent for a hackathon build. Server-side sessions are hashed at rest, rotate on password change, revoke on reset. Argon2id parameters follow OWASP. Every HTTP boundary uses zod and every WS event is validated against a discriminated union. Authorisation helpers run at request time with no caching (AC-6 is honoured). Drizzle usage is parameterised throughout; no `sql` template misuse was found. React escapes all user-controlled text.

However, three issues would fail an industrial review. First and foremost, the WebSocket `subscribe` handler in `ws/plugin.ts` accepts any topic the client names and has no authorisation check, so any authed user can read every room, DM, and user event stream in the system. Second, authentication endpoints are entirely unthrottled despite `@fastify/rate-limit` sitting in `package.json` – open season for credential stuffing. Third, password-reset tokens are logged at info level, so anyone with log access can take over accounts. A fourth, smaller-but-telling slip: the XMPP bridge calls `verifyPassword` with swapped arguments, so when the bridge is enabled nobody can authenticate (fails closed, but still wrong).

What's handled well: password storage, session hashing, membership + ban re-checks on every request, orphan attachment sweeping, SameSite=Lax cookies, content-addressed dedupe for files, and `Content-Disposition: attachment` on downloads.

## Findings

| Sev | Location | Issue | Why it matters | Fix |
|---|---|---|---|---|
| critical | `apps/api/src/ws/plugin.ts:85` | Client-initiated `subscribe` event subscribes the connection to any topic string with no ACL check. | Any signed-in user can `subscribe` to `room:<uuid>`, `dm:<uuid>`, or `user:<uuid>` for ids they have no business reading and receive every `message.new`, `message.updated`, `presence.update`, `invitation.received`, etc. published to that topic. Full confidentiality break on rooms and DMs. | Reject the client-side `subscribe` event entirely (auto-subscribe already covers every legitimate topic via `ws/auto-subscribe.ts` and the rooms/messages routes), or validate the topic: parse prefix, look up the uuid, and run `canAccessRoom` / `canAccessDm` / `userId === conn.userId` before calling `subscribeConnection`. |
| high | `apps/api/src/auth/routes.ts` (all five endpoints) | No rate limiting on `register`, `sign-in`, `password-reset/request`, `password-reset/consume`, `password-change`. `@fastify/rate-limit` is a dependency but never registered. | Credential-stuffing against `sign-in` is unconstrained. Password-reset request can be used to email-spam or user-enumerate by timing. Consume is guessable given the 32-byte entropy but there's no cost to try. | Register `@fastify/rate-limit` globally with tight limits on `/api/auth/*` (e.g. 5 attempts per IP per minute on sign-in, 3 on reset request/consume), per-ip for anonymous routes and per-user for authenticated ones. |
| high | `apps/api/src/auth/routes.ts:193` | Password-reset link (including the plaintext token) is logged via `req.log.info`. | Anyone with read access to app logs can trivially take over any account whose owner initiated a reset. In production this also ends up in any log aggregator. | Log only the user id or a redacted marker. Deliver the link to the user via whatever email sink is chosen – it must never pass through stdout. If a debug path is really needed, gate it behind `NODE_ENV==='development'`. |
| high | `docker-compose.yml:31` | `SESSION_SECRET` hardcoded to `"dev-only-session-secret-change-me-in-production-xxxxxxxxx"` in the delivery-contract file; same default lives in `apps/api/src/config.ts:11`. | The default is only cosmetically used today (cookies are not signed – see the info note below) but as soon as anyone flips `signed: true` on `reply.setCookie`, forgery becomes trivial. Also a red flag in any review. | Drop the default from `config.ts` (force the env var to be provided, `z.string().min(32)` without `.default(...)`), and either generate a secret on container start or document clearly that the compose file is dev-only. |
| high | `apps/api/src/xmpp/routes.ts:49` | `verifyPassword(row.passwordHash, parsed.data.password)` — arguments swapped vs the signature `verifyPassword(plain, hash)`. | When `ENABLE_XMPP_BRIDGE=1`, XMPP auth is effectively broken (fails closed, so not a disclosure issue), but no XMPP user can sign in. The fact the bridge still fails-closed is luck – if the argument order were reversed on the wrapper, this would be a silent auth bypass. | Swap to `verifyPassword(parsed.data.password, row.passwordHash)` and add a test. |
| medium | `docker-compose.yml:33` | `ALLOW_DEV_SEED: "1"` set in the committed compose file. | Lets any authed user hammer `/api/dev/seed-messages` to insert up to 20 000 messages per call – trivial DoS and disk filler. The dev-only endpoint is shipping live in the delivery contract. | Default to `"0"` in compose; flip it via an override file (`docker-compose.dev.yml`) when developers want seeding. |
| medium | `docker-compose.yml:11-12` | Postgres published on the host at `5432:5432` with credentials `app/app`. | On any machine that runs the stack with an open firewall the database is reachable from the network with trivial creds. Not a code bug but this is the delivery contract. | Drop the `ports:` stanza on `db` (internal docker network is enough) or bind to `127.0.0.1:5432` and rotate the credentials via env vars. |
| medium | `apps/api/src/server.ts:20-23` | No CSRF defence on state-changing POSTs / DELETEs. Cookies are `SameSite=Lax` which blocks cross-site top-level POSTs but not all cross-site contexts (cross-site child iframe GETs that mutate, a malicious same-site subdomain, or a shared-origin proxy). | In production the same-origin assumption via nginx mostly saves this – but there's no defence-in-depth. Combined with the SESSION_SECRET default, a sloppy deploy multiplies risk. | Add an `Origin` / `Referer` check on all mutating routes (or a double-submit cookie / CSRF token). `@fastify/csrf-protection` is the canonical choice. |
| medium | `apps/api/src/ws/plugin.ts:27-33` | WS upgrade accepts any `Origin`; authorisation comes only from the session cookie, which browsers will attach to any WS from any origin. | Classic cross-site WebSocket hijacking: a malicious page can open `wss://agora.host/ws` with the victim's cookie attached and ride the session. Given finding #1 above, they can also then subscribe to whatever they like. | Validate `req.headers.origin` against an allow-list on the upgrade hook and close with 4403 otherwise. |
| medium | `apps/api/src/server.ts:16` | No security response headers. nginx adds none either. | No CSP means any XSS escapes are unconstrained; no `X-Content-Type-Options: nosniff` means the browser may MIME-sniff attachment downloads; no `Referrer-Policy`, no `X-Frame-Options`. | Register `@fastify/helmet` with a conservative CSP (`default-src 'self'`, `img-src 'self' data:`) and add `nosniff` at nginx or via helmet. Add `X-Frame-Options: DENY` until the design confirms it's safe otherwise. |
| medium | `apps/api/src/auth/routes.ts:81-94` | Registration differentiates `email_taken` and `username_taken` with 409s. | Straightforward user/email enumeration – attackers learn which addresses and handles are registered. Sign-in itself uses a single generic `invalid_credentials` (good), but register re-opens the hole. | Collapse both 409 responses to a single generic `"already_taken"` code and perform both uniqueness checks before responding. Alternatively, accept the leak as a product trade-off and document it. |
| medium | `apps/api/src/attachments/routes.ts:77-78` | `originalFilename` and `mimeType` come straight from the multipart part with no validation or normalisation. | `mimeType` is later sent verbatim as the response `Content-Type` on download. Combined with no `X-Content-Type-Options: nosniff`, a user could upload a `.png` flagged as `text/html` and later trick other users into fetching it inline. The `Content-Disposition: attachment` header is currently saving the day, but depth matters. | Normalise the mime against a small allow-list (images → `image/png|jpeg|gif|webp|svg+xml`, everything else → `application/octet-stream`). Set `X-Content-Type-Options: nosniff` on the download response. Reject or sanitise filenames that contain path separators, control chars, or are longer than 255 bytes. |
| medium | `apps/api/src/auth/routes.ts:223-257` | `password-change` verifies the current password but does not rate-limit attempts. | Lets an attacker who's already hijacked a session (or sits next to an unlocked laptop) brute-force the current password for a locally authenticated user. | Add per-session attempt counters or use `@fastify/rate-limit` keyed by session id with a tight budget (e.g. 3 failures per 5 min, then locked out). |
| low | `apps/api/src/session/cookie.ts:13-21` | `reply.setCookie` is called without `signed: true`. The cookie value is a 32-byte random token (safe today), but the `SESSION_SECRET` default exists precisely for signing. | Today's behaviour is fine because the token itself is high-entropy and hashed in the DB. Flipping `signed: true` without also fixing the default secret would create a huge regression. | Either sign the cookie AND remove the default secret, or leave signing off and drop the default secret anyway (the cookie-plugin still wants a secret for tamper-evident API surfaces). Also consider `SameSite=Strict` on the session cookie – sign-in/sign-out are always same-origin POSTs from the SPA. |
| low | `apps/api/src/attachments/routes.ts:237-241` | `encodeRfc5987` only escapes `\`, `"`, newline, carriage return – but doesn't URL-encode non-ASCII. | A Russian-language filename (per product Russian-speaking users) will be inserted verbatim into the `Content-Disposition` header. Node may accept or reject; behaviour across proxies varies. | Use `filename*=UTF-8''<url-encoded>` per RFC 5987 and keep the ASCII `filename=` as a best-effort fallback (e.g. `filename="attachment"` for any non-ASCII case). |
| low | `apps/api/src/attachments/routes.ts:87` | `reply.code(500).send({ error: 'internal', message: (err as Error).message })` leaks raw error message. | Minor information disclosure – filesystem paths, drizzle internal messages, etc. may appear to clients. | Log the message server-side, return `{ error: 'internal' }` only. The other 500 at line 130 already does this. |
| low | `apps/api/src/messages/routes.ts:298` | Conversation list includes the last-message `body` even when the caller has an active user-ban with the DM counterparty. `canSendDm` is not re-checked for the preview. | Small confidentiality bleed: if A bans B, the sidebar preview for the A↔B DM still exposes the last message body from that thread. | Either hide the preview for DMs where `canSendDm` currently returns `banned`, or clear it on ban transition. |
| low | `apps/api/src/rooms/routes.ts:509-513` | `POST /api/rooms/:id/admins` reads `req.body` as `{ userId?: unknown }` without a zod schema. | Internal consistency – every other mutating route on this surface uses zod. A malformed `userId` that isn't a uuid will blow up later at the query level with a 500 rather than a 400. | Add a `zod` schema: `z.object({ userId: z.string().uuid() })`. |
| low | `apps/api/src/rooms/routes.ts:606-608` | `reason` field in the remove-member body read as `req.body.reason` with no length bound. | A caller can store a multi-megabyte reason in `room_bans.reason` (text column, no limit). Stored blow-up. | Tighten with `z.string().max(280)` like other reason fields. |
| low | `apps/api/src/ws/plugin.ts:46-68` | WS messages are parsed without size limits; an attacker could hold a connection open and flood huge JSON payloads. | `JSON.parse` on a multi-megabyte string blocks the event loop. Doesn't bring the process down quickly but is an easy DoS vector. | Configure `@fastify/websocket` with `maxPayload` (e.g. 64 KiB – well above message body cap of 3072 chars), and drop the connection on `maxPayload`-exceeded. |
| low | `apps/api/src/friends/routes.ts:485-519` | `GET /api/users/search` accepts any prefix `q` and returns up to 25 matches, no rate limiting. | Lets any authed user walk the user table prefix-by-prefix to enumerate all usernames in minutes. Usernames are public by product intent, but the cost to enumerate should not be zero. | Rate-limit the endpoint per user (e.g. 30/min) and require `q.length >= 2`. |
| low | `apps/api/src/session/plugin.ts:60` | `void touchSession(record.id).catch(() => undefined)` swallows the error silently. | If the DB update for session-slide fails, the session never refreshes its `expires_at`, and the failure is invisible. | info | `packages/shared/src/messages/index.ts:38` | `attachmentIds` accepted on `sendMessagePayload` but never linked to the stored message in `messages/ws-handlers.ts`. | Not a security bug per se, but means every attachment stays `messageId=NULL` and only the uploader ever sees it. Dead-code behaviour that hides intent. | Either implement the attach step inside the send transaction (with an ownership check on the attachment uploader) or remove the field from the schema. |
| info | `apps/api/src/dev/routes.ts:47` | `if (!allowed)` on `canAccessRoom` returns the bare `PermissionResult` for the dm branch, so the `allowed` check is truthy whenever `canAccessRoom` returns `{ ok: true }`. | The `canAccessRoom` branch is testing the object itself, not `.ok`. Happens to work because an object is always truthy. Not exploitable but the code is wrong. | Explicit `.ok` test on both sides. |

## Things done well (don't regress)

- Argon2id with OWASP 2026 parameters (`apps/api/src/auth/password.ts`).
- Session tokens are 32 bytes of `randomBytes`, SHA-256 hashed at rest (`apps/api/src/session/store.ts:37`). Never logged, never compared in cleartext.
- Password change and password reset both revoke every other session via `deleteSessionsForUserExcept` (`auth/routes.ts:219,255`).
- `Set-Cookie` is `HttpOnly`, `SameSite=Lax`, `Secure` in production (`session/cookie.ts`).
- Every mutating HTTP route except two bodies uses a zod schema; pattern is consistent across auth, rooms, friends, messages, attachments.
- WS events are validated against a discriminated union (`packages/shared/src/ws/events.ts`).
- Access checks re-run at every request – `canAccessRoom`, `canSendDm`, `isBannedFromRoom`, `canAccessDm` (see `messages/permissions.ts`, `rooms/permissions.ts`). CLAUDE.md's AC-6 rule is honoured.
- Account deletion relies on FK cascades with explicit `ON DELETE SET NULL` where history must survive, `CASCADE` for everything else – well-structured (`db/schema.ts`, migration 0000).
- Attachments are content-addressed, deduped, written via temp+rename (`attachments/storage.ts`).
- Orphan-attachment sweeper deletes DB row first then the file only if no other row references the hash (`attachments/sweeper.ts`).
- React renders all user content via `{value}` (no `dangerouslySetInnerHTML` anywhere) – no stored XSS path via message body, room name, or username as it stands.
- Drizzle is used for every query; no string-concatenated SQL, and the few `sql\`…\`` template usages (`auth/routes.ts:79`, `friends/routes.ts:504-511`) are parameterised correctly.
- Reset tokens are single-use with a conditional `UPDATE … WHERE consumed_at IS NULL` – safe against double-consumption races (`auth/password-reset.ts:60-66`).
- Password-reset token hashed at rest with `bytea`, matching session-token treatment.
- Multipart upload capped at `MAX_FILE_BYTES` and enforces `truncated` + image-specific cap after the fact (`attachments/routes.ts:90-104`).
- Attachment download ACL defers to the same permission helpers used by history fetch – single source of truth.
- Owner/admin moderation routes correctly reject self-removal and owner demotion.
- Friend-request idempotency + transactional accept (delete request + insert friendship) preserves invariants even under duplicate POSTs.

## Prioritised remediation order

1. Lock down WebSocket `subscribe`. Either drop it entirely or gate on the same permission helpers used by history fetch. Everything else is dwarfed by this.
2. Register `@fastify/rate-limit` and apply it to auth endpoints + the WS upgrade handshake.
3. Remove the password-reset token from the info log; gate any dev-time token echo behind `NODE_ENV==='development'`.
4. Tighten the delivery contract: no committed default `SESSION_SECRET`, no `ALLOW_DEV_SEED=1`, no host-exposed Postgres port.
5. Fix the swapped `verifyPassword` call in `xmpp/routes.ts` even if the bridge is off by default.
6. Add helmet (CSP + nosniff) and origin checking on `/ws`.
