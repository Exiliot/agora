# Security and privacy audit – agora

Date: 2026-04-19. Scope: full stack (web, api, shared, docker). Read-only audit; no code changed. Supersedes `docs/audits/security.md` (2026-04-18) where findings overlap; the older file is kept for trace.

## Executive summary

Security posture has moved forward materially since the 2026-04-18 round. The three critical / high issues that dominated the earlier audit are now closed: WebSocket `subscribe` is gated by a per-request ACL (`apps/api/src/ws/topic-acl.ts`), `@fastify/rate-limit` is registered and applied to every credential-handling endpoint (`apps/api/src/auth/rate-limit.ts`), and the default committed `SESSION_SECRET` is gone – the api refuses to boot with a stale default and generates a random one at startup otherwise (`apps/api/src/config.ts:40-47`). Argon2id parameters still follow OWASP, session tokens remain high-entropy-random and hashed at rest, and `@fastify/helmet` is now registered server-side alongside a strict CSP + security-header set in nginx.

Remaining issues are smaller but still shippable-risk. The top three in this round: (1) the reset-link token is still emitted to `stdout` via both `req.log.info` and a bare `console.error` at line 204 of `auth/routes.ts` – product-documented as a demo necessity but not gated on `NODE_ENV` so a production deployment would leak takeover tokens to the aggregator; (2) the WS upgrade handler does not check `Origin`, so cross-site WebSocket hijacking remains possible against a signed-in user even though the subscribe ACL now limits its reach; (3) the `/internal/xmpp/*` HTTP endpoints are completely unauthenticated – in the committed compose topology they are unreachable through nginx but any process on the docker network (including a compromised `web` container or an operator-added sidecar) can validate or enumerate passwords through them.

Password hashing, session revocation semantics, attachment ACL, server-side zod validation at every HTTP and WS boundary, drizzle parameterisation, and the absence of `dangerouslySetInnerHTML` across the React tree all look strong. FR-AUTH-13 cascade behaviour is implemented correctly via DB FKs, and forgotten-password tokens are single-use with a conditional `UPDATE` that survives races.

### Extended notes on the top findings

Three findings above benefit from a little extra colour before we walk the area-by-area notes:

**On finding 1 (reset link in logs).** The inline comment at `auth/routes.ts:197-201` is honest about the trade-off: in the demo there is no mailer, and the only way for an operator or tester to complete the reset flow is to read the link out of stdout. That is reasonable for a hackathon submission and is explicitly called out in FR-AUTH-9. The risk is only that a future operator wires the api to a real log aggregator (ELK, Loki, Datadog) without flipping `NODE_ENV` and the log line sails straight through with a valid takeover token for 30 minutes. A two-line change (gate both the `req.log.info` and the `console.error` behind `config.NODE_ENV === 'development'`) removes the risk entirely without changing the demo behaviour.

**On finding 2 (WS origin).** Without an origin check, any page the user visits can open a WebSocket to the agora deployment with the session cookie attached. `canSubscribeToTopic` confines what that socket can subscribe to, but the auto-subscribe logic runs as soon as `hello` is received, so the attacker's socket is subscribed to every room and DM the victim belongs to and will receive every `message.new` and `notification.created` event fanned out on those topics. The attacker can also send `message.send` events in those rooms as the victim. The subscribe-ACL mitigates confidentiality on *new* topics the attacker names, but does nothing for fan-out on auto-subscribed topics. Adding an `Origin` allow-list is a 10-line fix.

**On finding 3 (unauthenticated internal routes).** The routes are guarded only by `ENABLE_XMPP_BRIDGE=1` and the nginx path-gating. In the default `docker compose up` flow the flag is off and the routes are not even mounted – so the finding is zero-risk in the shipped topology. It becomes high-risk the moment an operator spins up the XMPP overlay on a machine with other tenants on the docker network, or if the nginx config drifts and `/internal/` accidentally gets proxied.

## Area-by-area notes

### 1. AuthN / AuthZ

- **Argon2id** with `m=65536, t=3, p=4` at `apps/api/src/auth/password.ts:8-13`. Matches OWASP 2026 guidance. Wrapper catches `argon2.verify` errors and returns `false` so corrupted hashes fail closed. Cost on a Node 24 container: roughly 60-120 ms per verify at those parameters, which sets a sensible floor for both the interactive sign-in path and any offline guessing against a stolen hash dump.
- **Session tokens**: 32 bytes of `randomBytes` at `apps/api/src/session/store.ts:50`, base64url-encoded, SHA-256 hashed before storage (`token_hash` as `bytea`). Unique index on the hash (`apps/api/src/db/schema.ts:76`). TTL 14 days sliding, pruned via `deleteExpired`. The session lookup is a single indexed probe per authenticated request; `session/plugin.ts` additionally throttles the sliding `UPDATE` to once per `SESSION_TOUCH_MIN_INTERVAL_MS` per session so the auth middleware does not turn every page navigation into a write. The in-memory `lastTouched` Map is not authoritative – if the process restarts, the first post-restart request slides the row, which is the desired behaviour.
- **Session cookie attributes** at `apps/api/src/session/cookie.ts:13-21`: `httpOnly: true`, `sameSite: 'lax'`, `secure: config.NODE_ENV === 'production'`, `path: '/'`, `maxAge` 14 days. Production `Secure` flag set correctly; dev over http without `Secure` is the right call. Not `signed` – safe because tokens are looked up by hash and have 256 bits of entropy (finding 12). `SameSite=Lax` blocks cross-site top-level POSTs but still permits the cookie on top-level GET navigations and on cross-site WebSocket upgrades (see finding 2).
- **Forgot-password tokens** at `apps/api/src/auth/password-reset.ts`: 32-byte random → base64url, SHA-256 hashed at rest with a `bytea` column (`apps/api/src/db/schema.ts:82-98`), 30-minute TTL, single-use via a conditional `UPDATE … WHERE consumed_at IS NULL RETURNING id`. Race-safe: two concurrent consumes of the same token race on `RETURNING` and only one wins. The `passwordResets` row is an insert-on-request; prior unconsumed rows for the same user are not invalidated, so a user who spams "forgot password" has up to N live tokens at once – acceptable given the per-endpoint rate limit (10/min) and the 30-minute TTL. Comparison is constant-time-ish: `eq(tokenHash, hashResetToken(token))` compares by full SHA-256 bytes, so timing leakage is within the btree lookup rather than a byte-wise compare.
- **Password-change** at `apps/api/src/auth/routes.ts:244-288` requires the current password (good) and invalidates sibling sessions via `deleteSessionsForUserExcept(user.id, session.id)`; revoked count is surfaced to the user via a `session.revoked_elsewhere` notification so they can spot a compromise. The endpoint is rate-limited at 10/min per IP – same budget as sign-in. No per-session attempt counter; an attacker with a stolen session cookie could burn 10 current-password guesses per minute per IP (finding 9).
- **Password-reset/consume** at `apps/api/src/auth/routes.ts:210-242` invalidates *all* sessions (pass `null` as keepId), matching the kill-everything posture required when credentials may be compromised. A post-consume `session.revoked_elsewhere` notification fires with `reason: 'password_reset'`.
- **Rate limiting**: `@fastify/rate-limit` registered in `rate-limit.ts` with global defaults off; `authRateLimit` (10/min) applied to `register`, `sign-in`, `password-reset/request`, `password-reset/consume`, `password-change`. Key strategy is IP-default; see finding 9 for the residual limitation. Store is the plugin's default in-memory map – not shared across api instances, but the MVP runs a single process (AC-SINGLENODE-1) so that is fine.
- **Enumeration resistance**: sign-in is correctly generic (`invalid_credentials` regardless of whether the email exists or the password is wrong). `password-reset/request` returns 204 in all cases without disclosing whether the email was registered. Register leaks `email_taken` vs `username_taken` (finding 6). Sign-in also handles the soft-delete case (`row.deletedAt`) by returning the same generic error, which is correct – a deleted user's email is otherwise free.

### 2. Permission / ACL at request time

- `canAccessRoom`, `canSendDm`, `loadDmForUser`, `canAccessDm` at `apps/api/src/messages/permissions.ts` run per request with no caching (AC-6). Each check is a direct DB query; there is no request-scoped memoisation or session-scoped cache, so a ban or membership change takes effect on the very next request.
- HTTP history fetch (`apps/api/src/messages/routes.ts:70-115`) correctly consults them for both room and DM branches. The DM branch composes `loadDmForUser` + `canSendDm` so a banned counterparty cannot read the history – tighter than some designs which would only require participation.
- `message.send` WS handler at `apps/api/src/messages/ws-handlers.ts:66-88` runs the same checks with full branch coverage. Denied sends emit an `err` with the permission code (`not_member`, `banned`, `not_friend`, `not_found`) plus a server-side `console.error` (finding 17).
- `message.edit` and `message.delete` at lines 229-313 skip the access check for the author case (finding 4). DM delete allows only author; room delete additionally allows owner/admin via a membership role check. The admin branch does correctly call `getMembership` (via the inline query at line 283-288) rather than assuming role from a cached value.
- `mark.read` at line 317 does not permission-check – it writes `last_read` regardless. A banned user could thereby reset their unread counter on a conversation they can no longer see. No data exposure, no fan-out to others. Low severity, not worth a finding.
- WS `subscribe` is ACL-gated at `apps/api/src/ws/topic-acl.ts`. Topic strings match the pattern `(user|room|dm):<uuid>`; anything else rejects. `user:<id>` requires `id === caller`. `room:<id>` requires a `roomMembers` row AND no `roomBans` row. `dm:<id>` requires that the DM exists and the caller is one of the two participants. Defensive: the ban check runs before the membership check for rooms, matching the schema invariant that ban removes membership.
- `/api/dev/seed-messages` correctly consults `canAccessRoom` / `canAccessDm`; the older 2026-04-18 finding about a shape bug has been fixed (it now destructures `.ok`).
- Room moderation routes (demote, remove, ban list, unban, delete) are each gated via `isAdminOrOwner` / `isOwner` with self-removal / owner-removal / admin-cannot-remove-admin rules at `apps/api/src/rooms/routes.ts:655-834`. Handler-level checks look correct. `admin_cannot_remove_admin` at line 725-727 correctly pulls `isOwner(id, callerId)` rather than trusting the earlier `isAdminOrOwner` result.
- Attachment download at `apps/api/src/attachments/routes.ts:169-195` consults the same permission helpers via `checkAccess` (line 209). Message ownership branch handles orphaned attachments (attachment with `message_id = NULL`) by restricting visibility to the uploader, and handles deleted messages by returning 404 to non-uploaders rather than leaking existence.
- DM user-ban semantics: once banned, history is preserved but no new messages can be sent – enforced at `canSendDm` via the `userBans` lookup with OR on both directions. Both sides keep the DM row and can read history (FR-FRND-6 says "history becomes read-only"). The banned side's sidebar preview body is blanked (`messages/routes.ts:321-323`); history fetch however requires `canSendDm` to pass, which is **tighter** than the spec – a banned user calling `GET /api/conversations/dm/:id/messages` receives 403, not the frozen history. This is a pre-existing behaviour not a regression; worth confirming against FR-FRND-6 with the product lead.
- Room ban semantics: `roomBans` removes membership in the same transaction (`rooms/routes.ts:737-751`). A banned user then fails `canAccessRoom` (ban check returns `banned` before the membership check). Access-lost event fires and the user's WS connections unsubscribe from the room topic via `publishRoomAccessLost`.

### 3. Input validation at boundaries

- Every HTTP handler reviewed routes its body through a zod schema from `@agora/shared` or a local zod schema; grep for `req.body` / `req.query` shows exactly these unvalidated readers and they all coerce or narrow before use:
  - `apps/api/src/rooms/routes.ts:199` – reads `req.query` as an inline `{ visibility?, q?, limit? }`. Validates `visibility === 'public'` and clamps `limit` via `Number.parseInt` + `Math.min(parsedLimit, 50)`. Safe but inconsistent with the rest of the codebase.
  - `apps/api/src/friends/routes.ts:209`, `:492`, `:553` – similar inline reads for `direction` and search `q`. All are validated in code before use; `direction` is checked against a literal set and `q` is trimmed and length-bounded.
- The `DELETE /api/users/me` body is ignored (no fields required). `POST /api/auth/sign-out` and `POST /api/auth/me` likewise read nothing from the body.
- WS envelope validation at `apps/api/src/ws/plugin.ts:81`: `clientToServerEvent.safeParse(parsed)` where `parsed` is `JSON.parse(raw.toString())`. On failure a `{type: 'err', payload: {code: 'validation'}}` frame is sent back. On success the discriminator type steers each handler.
- WS per-event payloads are trusted against the discriminator – the code-quality audit (finding 5) flagged this as shape-fragility; it is not a runtime security hole because zod's `discriminatedUnion` enforces every variant's payload shape. A future drift between `@agora/shared/ws/events.ts` and the handler cast would surface as a TS error first.
- Multipart upload (`apps/api/src/attachments/routes.ts:83-153`) enforces:
  - `fileSize: MAX_FILE_BYTES` at the plugin level (stream cuts off at this size).
  - `files: 1` (one file per request).
  - `fields: 4` (caps non-file fields – `comment` is the only currently-read one).
  - After write: re-checks `part.file.truncated || result.size > perFileCap`; if the per-type cap (image 3 MB, file 20 MB) is exceeded, returns 413 and purges the just-written file unless it dedupes an existing row.
  - Filename capped at 255 bytes.
  - Mime normalised to a 5-entry image allow-list or `application/octet-stream`.
  - Temp-file + rename via `writeStreamToStorage` avoids partial files at the canonical path.
  - SHA-256 computed during the write; dedupe is hash-based.
- Attachment download is strictly content-addressed by database id; path traversal is impossible because the filename on disk is `<hex-sha256>`, not the user-supplied `originalFilename`. `hashPath` asserts the hex hash is exactly 64 chars before building the path.
- Message bodies are validated via `messageBodySchema` (`packages/shared/src/messages/index.ts:13-20`): `min(1)`, `max(3072)` chars, and a `.refine` that re-checks the UTF-8 byte length is ≤ 3072. Prevents the classic "3000-char body of 4-byte emoji" bypass against a char-length cap.
- `sendMessagePayload` limits `attachmentIds` to `max(4)`. Room names and descriptions have their own zod bounds in `@agora/shared/rooms`. User bans accept an optional `reason` string (via `userBanCreate`) – worth confirming its max length matches the friend-request note and room-ban reason limits.

### 4. SQL injection / drizzle raw SQL

- All `sql\`…\`` usages reviewed (grep results summarised):
  - `apps/api/src/notifications/publisher.ts:43-53` – parameterised with `${newId}`, `${args.userId}`, etc. Safe.
  - `apps/api/src/messages/routes.ts:266-287` – uses `sql.join` of `sql\`(${'room'}::conversation_type, ${r.id}::uuid)\`` fragments; `r.id` is a UUID string pulled from the caller's own memberships and is bound, not concatenated. Safe.
  - `apps/api/src/friends/routes.ts:572` – `sql\`lower(${users.username}) like lower(${\`${rawQ}%\`})\``. `rawQ` is bound as a parameter via the template literal substitution; drizzle emits `$1` for the inner `${...}`. The outer template-string concatenation `${rawQ}%` is an application-level concern but safe because the result is bound as a single scalar. The ILIKE itself is the only behaviour risk and is expected.
  - `apps/api/src/auth/routes.ts:83,92,134,190` – `lower(${users.email}) = lower(${email})` patterns with `${email}` bound. Safe.
  - `apps/api/src/xmpp/routes.ts:58,72` – `lower(${users.username}) = ${local}` with `local` bound. Safe.
  - Schema-level `sql\`…\`` usages (check constraints, partial index predicates) are static and do not involve user input.
- No string concatenation inside a `sql\`\`` template was found.
- Drizzle `db.execute<T>(sql\`…\`)` return rows correctly typed; `notifications/publisher.ts` and `messages/routes.ts` are the only raw-execute sites and both use the parameterised form.

### 5. Information leakage

- Error shapes are inconsistent across features (see code-quality audit finding 12) but none of them leak sensitive internals except `apps/api/src/attachments/routes.ts:106` which surfaces the raw error message on the upload-500 path (finding 13).
- Reset link in logs is documented in the comment at `auth/routes.ts:197-201` as a demo necessity. Still a production risk (finding 1).
- No observed log lines include `password`, session token, or session token hash. `console.warn` / `console.error` usages in `ws/connection-manager.ts`, `attachments/sweeper.ts`, `attachments/bus-handlers.ts` etc. log error objects and identifiers but no credentials.
- `/api/users/search` filters out `deletedAt IS NOT NULL` and hides users who have banned the caller (`NOT EXISTS (SELECT 1 FROM user_bans WHERE banner_id = users.id AND target_id = caller)`). Good. Enumeration cost is limited by the 30/min rate limit and the 2-char minimum query; still walkable (finding 10). Note the asymmetry: the search hides users who banned the caller (so a banned user cannot re-discover the banner via search), but does NOT hide users the caller has banned – presumably so the caller can find and unban them. Worth confirming this is intentional.
- `/api/users/:username` (profile lookup) returns `{id, username, createdAt}` for any registered, non-deleted user. It does not filter against bans, so a banned user can still fetch their banner's profile page. Product likely fine; flag for awareness.
- Notifications `payload_json` contains short previews of the message (120-char `snippet`), sender/inviter/banner usernames, and ban reasons. Each row is scoped to `userId = <recipient>` and recipients are filtered via `and(eq(notifications.userId, req.user.id), …)` on every read path. No cross-user leakage observed. One nuance: notifications for room bans include the `reason` field which an admin wrote – that's delivered to the banned user, matching the product expectation (FR-ROOM-11 lets admins see who banned each banned user, and the target naturally learns the reason from the notification).
- DM preview under ban blanks the body but exposes the `createdAt` and `authorId` (finding 8). The room sidebar does not do the same filtering because room messages aren't subject to the user-ban rule.
- 404 vs 403 distinction: `canAccessRoom` returns `not_found` for rooms that don't exist, `not_member` or `banned` for rooms that do exist but are unreachable. The history route maps `not_found` to 404 and everything else to 403. A banned user therefore cannot distinguish "room deleted" from "I'm still banned" on the history endpoint; room detail (`GET /api/rooms/:id`) does return 403 for banned and 404 for non-existent, which is the correct pair.
- Private-room catalogue: `GET /api/rooms?visibility=public` only returns public rooms. Private rooms are never enumerable via this path. `GET /api/rooms/mine` correctly only returns rooms the caller is a member of, so private rooms do not leak via the member-count subquery.

### 6. Transport + headers

- SPA served by nginx (`apps/web/nginx.conf`). Headers:
  - `Content-Security-Policy`: `default-src 'self'`, `script-src 'self'` (no inline, no `'unsafe-eval'`, no CDN), `style-src 'self' 'unsafe-inline' https://fonts.googleapis.com`, `font-src 'self' https://fonts.gstatic.com`, `img-src 'self' data:`, `connect-src 'self'`, `frame-ancestors 'none'`, `base-uri 'self'`, `form-action 'self'`. Strong. The `style-src 'unsafe-inline'` carve-out is driven by the design-system's inline `style={}` usage – tracked in the design docs; an acceptable trade-off given there is no user-injectable style path.
  - `X-Content-Type-Options: nosniff` – prevents the browser from second-guessing attachment MIME types.
  - `X-Frame-Options: DENY` – clickjacking defence (also covered by CSP `frame-ancestors 'none'`).
  - `Referrer-Policy: strict-origin-when-cross-origin`.
  - `Permissions-Policy: geolocation=(), microphone=(), camera=(), payment=()` – disables features the app never requests.
- No `Strict-Transport-Security` header – acceptable for a localhost demo; needs adding if the app is deployed over HTTPS.
- API side: `@fastify/helmet` registered with `contentSecurityPolicy: false` (finding 16) and `crossOriginEmbedderPolicy: false`. Default helmet defaults still apply (`X-Content-Type-Options`, `Referrer-Policy`, `Cross-Origin-Resource-Policy`).
- No `@fastify/cors` registered – the api has no CORS headers. This is correct for the shipped topology (web and api share the nginx origin). External browser clients not served by the same nginx will simply fail pre-flight, which is the desired outcome. Flagged to any future operator tempted to add `cors({ origin: '*' })`: doing so would combine badly with cookies-based auth.
- Websocket `/ws` accepts any origin (finding 2). No origin allow-list in `apps/api/src/ws/plugin.ts`. The `@fastify/websocket` plugin has a `verifyClient` hook available which is the natural place to enforce this.
- `trustProxy: 1` in `server.ts:22`: Fastify trusts the first hop's `X-Forwarded-For`. nginx at port 8080 is the expected proxy. If the api is ever exposed directly (not through the committed nginx), `req.ip` would become whatever the client claims – combined with IP-keyed rate limiting this would allow trivial bypass. Flagged for review at deployment time.

### 7. Privacy

- **PII stored**: email, username, password hash, `sessions.ip` + `sessions.user_agent`, `users.created_at` / `updated_at` / `deleted_at`. `users.password_hash` never leaves the row. `sessions.token_hash` is an SHA-256 digest, never plaintext. Messages and attachment metadata contain content the user themselves produced and can be considered first-party. `notifications.payload_json` stores denormalised snippets and counterpart usernames; see area 5 above.
- **`DELETE /api/users/me`** at `apps/api/src/auth/routes.ts:290-306` relies on FK cascades. Per the schema:
  - `sessions.user_id` → `ON DELETE CASCADE` (session rows gone).
  - `rooms.owner_id` → `ON DELETE CASCADE` (owned rooms + their members/messages/attachments gone per cascading FKs).
  - `room_members.user_id` → `ON DELETE CASCADE` (memberships gone).
  - `friendships.user_a_id`, `friendships.user_b_id` → `ON DELETE CASCADE`.
  - `user_bans.banner_id` / `.target_id` → `ON DELETE CASCADE`.
  - `friend_requests.sender_id` / `.recipient_id` → `ON DELETE CASCADE`.
  - `messages.author_id` → `ON DELETE SET NULL` (matches FR-AUTH-13: "messages they authored in other rooms remain").
  - `attachments.uploader_id` → `ON DELETE SET NULL` (same reasoning).
  - `room_bans.banner_id` → `ON DELETE SET NULL` (ban row persists, banner anonymised).
  - `room_invitations.inviter_id` → `ON DELETE SET NULL`.
  - `notifications.user_id` → `ON DELETE CASCADE`; `notifications.actor_user_id` → `ON DELETE SET NULL`.
- **Post-cascade surfaces**: `hydrateMessages` (`apps/api/src/messages/history.ts:66-103`) handles `authorId === null` gracefully by omitting the author in the `MessageView`. The UI renders a "(deleted user)" placeholder. Matches FR-AUTH-13. Same pattern applies to attachment uploader and room-ban banner displays.
- **Notifications after deletion**: `notifications.actor_user_id` → `ON DELETE SET NULL`, so prior "X invited you" / "X mentioned you" bell entries persist without an actor-id after the actor deletes their account. The hydrate path at `notifications/hydrate.ts` tolerates a null actor cleanly (returns `actor: null`). The payload_json however may still contain the actor's username as a frozen string – a deleted user's username therefore survives in notification payloads until the notification row is itself deleted. Acceptable given the spec's set-null posture, but worth knowing.
- **GDPR-like export**: no export endpoint exists. Per product brief this is out of scope; noted as a known gap. A manual export path exists via SQL: `users`, `sessions`, `messages WHERE author_id = ?`, `attachments WHERE uploader_id = ?`, `friend_requests WHERE sender_id = ? OR recipient_id = ?`, `friendships WHERE user_a_id = ? OR user_b_id = ?`, `notifications WHERE user_id = ?`, `last_read` and `conversation_unreads` by user id would cover the full PII footprint.

### 8. Attachments + storage

- Uploaded files under `/data/attachments/<hash[0:2]>/<hash[2:4]>/<hash>`; docker volume `attachments` (line 46 of compose). Owned by the api container's user (node:24-alpine default uid). No other service mounts the volume.
- Content-type: now normalised against a small image allow-list (`png`, `jpeg`, `gif`, `webp`, `avif`); everything else becomes `application/octet-stream`. Combined with `X-Content-Type-Options: nosniff` (line 191) and `Content-Disposition: attachment` (line 192), an uploaded `.html` or `.svg` cannot be rendered inline. Improvement since 2026-04-18.
- SVG is deliberately *not* in the image allow-list – this matters because SVG is the classic vector for stored XSS (embedded `<script>` tags execute when rendered inline). Uploading an SVG today results in `mimeType === 'application/octet-stream'`, which the browser cannot render inline even if CSP were relaxed.
- Download ACL delegates to `checkAccess` which uses `canAccessRoom` / `canAccessDm` – same helpers used elsewhere (single source of truth, good). The orphan branch at line 213-218 correctly restricts visibility to the uploader.
- `Content-Disposition` helper (`contentDispositionFor` at line 256) now emits both ASCII fallback and RFC 5987 `filename*=UTF-8''…`. Cyrillic and emoji filenames round-trip. Improvement since 2026-04-18.
- **Content-hash deduplication**: two users uploading the same bytes share a single file on disk but distinct `attachments` rows with their own `uploader_id`, `message_id`, `comment`, and `original_filename`. The orphan sweeper at `attachments/sweeper.ts:20-53` only deletes the physical file after confirming no other live row references the hash – correct ordering to avoid dangling references.
- **Room deletion cascade**: `attachments/bus-handlers.ts` wraps `bus.publish` to react to `room.deleted` events and delete the underlying files. The wrapper pattern is fragile (see code-quality audit finding 8-10) but security-correct: it removes the files *after* the rooms transaction has deleted the messages rows, so the dedup check correctly sees the refcount drop to zero.
- **Upload limits**: per request `MAX_FILE_BYTES` (20 MB non-image, 3 MB image), one file per multipart, four fields max. A user can still upload many files sequentially – there is no per-user quota. At hackathon scale that is fine; in production a quota plus a sweeper watchdog would be needed to bound disk use.

### 9. XMPP bridge

- ADR-0005 sidecar. `ENABLE_XMPP_BRIDGE` defaults to `'0'` (config.ts:15), so `/internal/xmpp/*` routes are not mounted in the default `docker-compose up` flow. The XMPP compose overlay (`docker-compose.xmpp.yml`) is opt-in via `docker compose -f docker-compose.yml -f docker-compose.xmpp.yml up`.
- When enabled, the routes carry no authentication (finding 3). `verifyPassword` argument order is now correct (`parsed.data.password, row.passwordHash` – fixed since 2026-04-18).
- Usernames are compared case-insensitively (`lower(username) = local`) – matches the registration rule and avoids mixed-case bypass.
- No rate-limit on the check_password route, so a compromised sibling container is bounded only by argon2 verify cost.
- Prosody federates via XEP-0220 dialback (no pre-shared certs), per the implementation notes in ADR-0005. This is appropriate for the demo federation but not for production; worth spelling out in an operator-facing note if the bridge is ever shipped.
- Response body shape (`true` / `false` plaintext for the GET routes, JSON `{ok, userId}` for the POST) matches the `mod_auth_http` contract; a misbehaving Prosody that silently accepts any response would bypass this, but that's a Prosody-side concern.

### 11. WebSocket hardening detail

A dedicated section because the WebSocket surface underwent the most change since the 2026-04-18 audit and deserves a careful walk-through:

- **Upgrade auth**: `apps/api/src/ws/plugin.ts:29-35` closes the socket with app-code `4401` if `req.user` is null. The session plugin runs as an `onRequest` hook and has already validated the cookie (indexed lookup on `token_hash`, plus user-row `deletedAt` check), so by the time the `/ws` handler fires the user is fully resolved. A user whose session is revoked mid-upgrade will be closed. Good.
- **Live revocation**: subsequent operations (mark.read, message.send, etc.) all go through `canAccessRoom` / `canSendDm` which re-hit the DB. A user banned *after* the WS is established continues to hold their open socket (no active disconnect) but cannot act on the banned room. The `room.access_lost` bus event triggers the client to unsubscribe and navigate away; the server does not need to tear the socket down.
- **Heartbeat + AFK**: ws/plugin.ts schedules a 30 s ping interval with a terminate-on-missed-pong. This is transport-level keepalive, not auth – an attacker in the middle cannot use it to prove liveness.
- **Subscription lifecycle**: on `hello`, `auto-subscribe.ts` walks `room_members` and `dm_conversations` and subscribes the connection to each. On `close`, `unsubscribeAll` tears every subscription down. There is no re-authorisation on reconnect beyond the session cookie; a user banned from a room while disconnected will fail the subscription check on the next `hello` only if `canSubscribeToTopic` is invoked – which it isn't, because auto-subscribe goes straight to `subscribeConnection`. This is safe because auto-subscribe consults the authoritative DB (`room_members`) for the user's current memberships, so a banned user's membership row is already gone.
- **Topic ACL**: `canSubscribeToTopic` is invoked only on client-initiated `subscribe` events, not on auto-subscribe. That's correct – the auto-subscribe path uses authoritative DB rows, so double-checking would be redundant. The hardening lives where the untrusted input is (client-supplied topic strings).
- **Outbound back-pressure**: `connection-manager.ts:23-48` bounds the outbound queue at 512 frames and drops `presence.update` events under pressure, never drops messages. The `queued` counter can leak on a dead socket (code-quality audit finding 33) – mild reliability risk, not a security issue.

### 12. Threat-model sketch

For completeness, the threat actors and the positions they could take against the deployment:

- **Unauthenticated external attacker.** Can hit `/api/auth/*`, `/health`, and `/ws`. Bounded by rate limits on auth endpoints. Cannot mount credential-stuffing beyond 10 attempts/min per IP. Cannot read any conversation. Cannot cause notable DoS without a proxy pool.
- **Authenticated user, benign.** Expected baseline. Can read their own conversations, DM their friends, upload attachments within the per-file cap, create rooms. Rate limits and permission checks behave as designed.
- **Authenticated user, malicious.** Can attempt:
  - Subscribing to foreign topics via client-side `subscribe` events – blocked by `canSubscribeToTopic`.
  - Hitting `/api/attachments/:id/download` with foreign ids – blocked by `checkAccess`.
  - Walking the user directory via `/api/users/search` – rate-limited and bounded at 750 matches/min per IP.
  - Reading a foreign user's profile via `/api/users/:username` – 200 OK but exposes only `{id, username, createdAt}` which is product-public.
  - Harvesting notifications – scoped to `user_id = self`, no leakage.
  - Spamming messages – bounded by body-size zod schema but not per-user rate-limited; worth noting.
- **Cross-site attacker via victim's browser.** SameSite=Lax blocks top-level cross-site POSTs. WS upgrade from a cross-site page remains possible (finding 2). A `form-action 'self'` CSP on the SPA prevents cross-site form POSTs from the agora origin.
- **Compromised sibling docker container on the same compose network.** Can reach `api:3000` directly, bypassing nginx. If the XMPP bridge is enabled, can enumerate users and crack passwords at argon2 speed (finding 3). Otherwise can hit the public API with no cookie and thus no session – so only the unauthenticated auth routes are reachable.
- **Compromised host with log read access.** Can harvest reset tokens (finding 1). Cannot decrypt session tokens from the DB (hashed) or steal live sessions from stdout (never logged).
- **Compromised database.** Password hashes are argon2id – an offline crack is costly per attempt. Session tokens are SHA-256 hashed – an attacker with a DB dump cannot issue session cookies.

### 13. Additional observations

A few cross-cutting notes that do not map neatly to the ten scoped areas but are worth capturing before the next round:

- **Error handler** at `apps/api/src/server.ts:44-55` returns a uniform `{error, message}` shape and suppresses internal error messages on 5xx (`message: 'internal error'`) while logging the underlying `err` server-side. Good fallback. Specific handlers still throw their own bespoke shapes, which is the inconsistency flagged in the code-quality audit; it is not a security hole.
- **Health endpoint** at `server.ts:57-65` returns `{status, db, ts}`. No authentication required – that's correct for a liveness probe. The payload does not expose anything useful to an attacker beyond "the app is up".
- **Graceful shutdown** at `server.ts:70-85`: wraps `app.close` and `pool.end` in try/catch and exits non-zero on failure. Improvement since the code-quality audit round.
- **No secrets in the repo**: grep for `password`, `secret`, `token`, `key` across the tree returns only the legitimate schema references, test fixtures, and the `SESSION_SECRET` env var documentation. No committed credentials.
- **Session binding to IP / UA**: the session row records the first X-Forwarded-For hop and the User-Agent string at creation, but the middleware does not re-check them on subsequent requests. This is the correct call for the product (users roam between wifi / mobile networks) but worth noting in the sessions ADR as an explicit non-feature.
- **Idempotent transactional writes** are used consistently for the critical paths: friendship accept (delete request + insert friendship), user-ban (delete requests + delete friendship + insert ban), room-remove (insert ban + delete membership). All guarantee the ban-row + membership-row invariants that `canAccessRoom` and `canSendDm` depend on.

- **Shared zod schemas** in `@agora/shared` keep the wire contract in one place. The `clientToServerEvent` discriminated union at `packages/shared/src/ws/events.ts:59-69` is the entry point for every WS message. Adding a new client event requires updating the union, at which point every handler is type-narrowed against the new payload.

- **No build-time magic** that could introduce security surprises: Vite bundles the SPA, tsx runs the api directly, drizzle-kit generates migrations from the schema. There is no eval-at-runtime code generation, no dynamic import of user input, no template engine consuming user-supplied strings.

- **The tests folder** at `apps/api/tests/integration/` mocks the DB rather than spinning up Postgres. This is a test-quality concern (code-quality audit finding 17) not a security concern, but it does mean the permission helpers are not exercised against real FK cascades and ban-row invariants in CI. A future audit round should prioritise standing up a testcontainers-backed integration layer.

- **Input length invariants** are worth codifying. Room name, room description, username, email, message body, reset token, session cookie – all have explicit bounds in the shared schemas. Attachments add a 255-byte filename cap inline. The two missing bounds noticed during the audit are (a) `userBanCreate.reason` (check it matches the 280 char pattern used elsewhere) and (b) `removeMemberBody.reason` which already has 280. A one-shot audit that asserts every free-text field has a documented upper bound would close out this category for good.

- **No console output or response body observed** leaks the session cookie, session token, password, password hash, or the raw contents of `payload_json` to unintended recipients. The audit grepped for each of these terms across the api source tree with no hits outside of their legitimate storage and fetch paths.

## Closed since 2026-04-18

The earlier audit's three critical / high findings and several mediums are resolved on current `main`:

- **WebSocket `subscribe` ACL** – gated by `canSubscribeToTopic` at `apps/api/src/ws/topic-acl.ts`, called from `apps/api/src/ws/plugin.ts:116-131`. Unknown topic formats, non-matching user topics, non-member room topics, and non-participant DM topics all reject with a `forbidden` err.

- **Auth rate limiting** – `@fastify/rate-limit` registered at `apps/api/src/auth/rate-limit.ts` and applied to all five credential endpoints (`register`, `sign-in`, `password-reset/request`, `password-reset/consume`, `password-change`). IP-keyed, 10/min.

- **Committed `SESSION_SECRET` default** – removed from `config.ts`. Optional in env; if unset, a random secret is generated at boot with a warning logged. `docker-compose.yml:33-34` comments the env var out intentionally.

- **`ALLOW_DEV_SEED=1` in the shipped compose file** – removed; defaults to `'0'`, comment at line 40-42 of compose calls out that it should be enabled via an override file only.

- **Postgres port published on 5432** – removed; the `db` service no longer exposes a host port. Comment at line 11-14 of compose explains.

- **XMPP `verifyPassword` argument order** – fixed at `apps/api/src/xmpp/routes.ts:62` (`verifyPassword(rawPass, row.passwordHash)`).

- **Missing helmet / security headers** – `@fastify/helmet` registered at `apps/api/src/server.ts:28`, plus the full header set on the SPA in `apps/web/nginx.conf:11-15`.

- **Multipart mime sniff** – `normaliseMime` allow-list at `apps/api/src/attachments/routes.ts:33-50`. `X-Content-Type-Options: nosniff` added on the download response at line 191.

- **Filename UTF-8 encoding** – `contentDispositionFor` at line 256 emits RFC 5987 `filename*=UTF-8''…` with an ASCII fallback.

- **DM preview under ban** – `messages/routes.ts:321-323` blanks the preview body when either side has user-banned the other.

- **`POST /api/rooms/:id/admins` body validation** – now parsed via `adminPostBody` (`rooms/routes.ts:21`).

## What's already solid

Argon2id with OWASP 2026 parameters and a dedicated spec file pinning the argon2 parameter string. Server-side session model backed by a hashed token column with a unique index and explicit sliding-expiry throttling. Every mutating HTTP route is zod-validated at the boundary via shared schemas (`@agora/shared`), and every WS event goes through a discriminated union before it reaches a handler. Permission checks run per request with no caching or memoisation, per AC-6. Drizzle is used exclusively for every query; raw `sql\`\`` templates are parameterised without exception. React renders user content through `{value}` only – no `dangerouslySetInnerHTML`, no dynamic HTML string construction, and the SPA's CSP forbids inline scripts. FR-AUTH-13 cascade behaviour is structurally correct via schema-level FK policies (CASCADE vs SET NULL chosen deliberately per history-retention rule). Session revocation semantics are immediate – password-change keeps the current session and kills siblings, password-reset kills everything, individual session revoke is a direct DB delete. The attachment pipeline is content-addressed with dedupe, temp+rename writes, an orphan sweeper that deletes DB rows before bytes, and a download path that reuses the same permission helpers as the main messaging API. The WS topic-ACL layer is the single most meaningful security improvement since 2026-04-18 and it is well-factored.

## Recommended ADRs

Three architectural pieces deserve pinning while the rationale is fresh, so the next audit round does not re-open settled debates:

- **ADR-0006 · WebSocket origin + subscribe ACL.** Rationale: the `/ws` upgrade currently accepts any origin and relies on the cookie; combined with the subscribe ACL this is mostly safe, but an `Origin` allow-list is cheap defence-in-depth and clarifies the deployment expectation that the SPA and api share a nginx origin (or a small trusted set). The ADR should name the allow-list construction (use `APP_BASE_URL` by default, extend via an optional `WS_ORIGIN_ALLOWLIST` env var), the close code used on rejection (`4403`), and the behaviour when `Origin` is absent (Node WS client with no origin header – decision: reject, because real browsers always send it).

- **ADR-0007 · Reset-link delivery posture.** Rationale: the spec requires a mocked email, FR-AUTH-9 requires the link to be visible "in the server log for testability", and the current impl logs the full takeover URL at info level. An ADR nails down which logger calls are demo-only (gated on `NODE_ENV === 'development'`) and records the future migration path when a real mailer is wired in. It should also name the acceptable footprint during demo runs: stdout only, never aggregated, never stored.

- **ADR-0008 · Rate-limit keying strategy.** Rationale: the per-IP default leaves `sign-in` open to credential-stuffing from proxy pools. An ADR captures the deliberate choice to compose keys (`ip` + `email` for sign-in and reset-request, `session.id` for password-change, `user.id` for search) and names the acceptable false-positive rate for legitimate NATed users. It should also cover the implications for `/internal/xmpp/auth` if that path ever opens up.

- **ADR-0009 · Internal-only HTTP surface.** Rationale: `/internal/xmpp/*` is currently mounted on the same Fastify listener as the public API, relying on nginx path-gating for protection. An ADR should record whether to bind a second listener to an internal port, introduce a shared-secret header, or keep the current posture with a documented assumption. Pick one and pin it.
