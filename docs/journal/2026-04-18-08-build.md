# 2026-04-18 · Build day

Continues `2026-04-18-01-kickoff.md`. Covers the actual implementation work from ~12:00 to end-of-day on the Saturday track.

## How the work was organised

Two tiers of workers, operating in parallel on a single `main` branch:

- **Coordinator (main session)** owned the shared infrastructure that every feature depends on — DB schema, migrations, in-process bus, WS connection manager, session store, route registry, design-system primitives, routing, app shell, integration hygiene.
- **Subagents** (worktree-isolated) built self-contained feature modules in parallel. Three parallel agents at a time, each on its own git worktree branch, each committing feature-specific work. The coordinator merged each branch back into `main` sequentially, resolving the one predictable conflict (`apps/api/src/routes/index.ts` — each feature adds one import line) on the way in.

Phases:

1. **Phase 1 · Shared infrastructure (coordinator solo)** — Drizzle schema for all 15 tables, startup migrator, in-process pub/sub bus with `room:` / `dm:` / `user:` topic conventions, WS connection manager + dispatcher + route registry, Drizzle-backed session store with Argon2id passwords.
2. **Phase 2 · Backend features (three parallel subagents)** — auth + sessions, rooms, messages. All three merged cleanly. One bug in the session store's "log out all others" helper (used `lt` instead of `ne`) was flagged by the auth agent and patched on merge.
3. **Phase 3 · Backend features (three parallel subagents)** — presence, friends, attachments. All three merged cleanly. The attachments agent couldn't subscribe wildcard to the bus (no API for it), so it monkey-patched `bus.publish` to intercept `room.deleted` events. Pragmatic.
4. **Phase 4 · Frontend (coordinator solo)** — design system ported verbatim from the Claude Design handoff (18 components), reconnecting WS client, typed API client, full routing shell, auth pages, chat view with sidebar + composer + context panel, public room catalogue, active sessions view, contacts page with friend requests + bans + invitations, DM view, manage-room modal.
5. **Phase 5 · Integration (coordinator solo)** — six end-to-end Playwright tests covering health, wordmark, single-user room creation, two-user real-time, and friends → DM round-trip.

## Bugs caught by e2e that typecheck missed

Four distinct issues surfaced only when the real stack ran in a browser:

1. **nginx `proxy_pass` trailing slash** stripped `/api/` from the forwarded path. Backend routes are mounted at `/api/auth/register` etc.; nginx was forwarding `/auth/register`. Dropped the trailing slash so the prefix is preserved.
2. **Web build shipping the old `App.tsx`** — the earlier Write call of the new App.tsx (with Router, providers, etc.) somehow didn't persist on disk despite the tool reporting success. Discovered via Playwright rendering the original scaffolding page. Re-written and shipped.
3. **WS auth gate read the wrong property** — `extractUser` was looking for `req.session.user`, but the session plugin attaches `req.user` separately. Every WS upgrade was closing with 4401 and the client reconnected in a tight loop until corrected.
4. **DM auto-subscribe on open** — when a DM is created, neither participant's WS is subscribed to the new `dm:<id>` topic until they reconnect. Added a fan-out that subscribes all live connections for both users at DM creation.

The nginx one is the most interesting: curl directly at port 3000 bypasses nginx and hides the bug. The delivery contract (`docker compose up` and hit the web port) is the only test that catches it. Yet another argument for `/smoke` after every significant change.

## Architecture choices that paid off

- **Topic-based pub/sub with `room:` / `dm:` / `user:` prefixes.** Every feature wires in by publishing to the right topic; the WS layer maintains per-connection subscription sets and forwards. One `Bus` implementation, a hundred feature events.
- **Server-side sessions in DB with sliding expiry.** Cookie auth Just Works for both HTTP and WS upgrade — no JWT parsing, no token rotation story. Banning a user mid-session drops their session row and they're out.
- **Two-agent-tier parallelism with file-boundary discipline.** Every feature module lives in its own subdirectory. The only shared file agents touch is `apps/api/src/routes/index.ts`, and only to add one import line. Merge conflicts were predictable and mechanical.
- **Claude Design handoff → tokens → TS components** — the visual language (warm paper, IBM Plex Mono, no bubbles, hairlines > shadows) survived the port verbatim. Zero bikeshedding mid-event.

## What we chose not to ship

Per ADR-0005 and scope discipline:

- **XMPP/Jabber federation (ST-XMPP-*)** — sidecar approach preserved as Phase 2 plan. Room/user identity already JID-compatible so a future Prosody sidecar drops in without schema changes.
- **Attachments UI** — backend done (upload, download, ACL, orphan sweep, cascade on room delete). Frontend composer doesn't expose paste/upload yet. Would take another ~60–90 min but not required to demo the chat product.
- **Presence indicators wired in the UI** — backend running, but the sidebar shows everyone as "offline" visually. Real-time status propagation works at the protocol level but the client doesn't subscribe to `presence.update` and render.
- **Message edit/delete in the UI** — backend handlers exist; UI just doesn't surface them.
- **Notification bell with counts** — friend requests and invitations appear as sections on `/contacts`, not as a top-nav popover.

## Takeaways

- **Documentation-first paid off enormously.** When a subagent started with a cold context, the spec, architecture, data model, WS protocol, ADRs, and design bundle gave them a complete brief. Every agent's report confirmed they used the docs. The "context before code" philosophy stopped being a meta-point and started showing up as less rework.
- **Worktree isolation is a mixed blessing.** Pros: independent branches, no conflicts on the writable files. Cons: no `node_modules`, so agents can't typecheck unless they symlink. The rooms and friends agents figured this out and left symlink-then-remove snippets in their journals. Worth baking into CLAUDE.md for future sessions.
- **The coordinator owns the shared rails; subagents own the trains.** Tried to enforce "don't touch server.ts, don't touch shared package". Both held. A single collision point (`routes/index.ts`) was manageable because every merge knew what the conflict looked like and how to resolve it.
- **Test the hardest flow first.** Two-user real-time caught two bugs that single-user never would have. Rule for next time: start e2e with the most complex scenario.
- **`exactOptionalPropertyTypes: true` bites in specific places** — `transport: undefined` in the Fastify logger config, `body: undefined` in `fetch` init, `color: undefined` in a React prop. The pattern is always the same: pass a partial object when the field is absent, don't stamp `undefined`.
