# 2026-04-18 · Presence feature module

Implementation of the in-memory presence state machine (ADR-0003) and the WS
wiring that drives it. Built in an isolated worktree in parallel with the other
feature agents.

## Scope delivered

- `apps/api/src/presence/registry.ts` — the `Map<userId, Map<tabId, {lastActivityAt}>>`
  registry with `registerTab`, `recordActivity`, `removeTab`, `scheduleRemoveTab`,
  `computeStateFor`, `listTrackedUserIds`, `hasUser`, and `getSnapshotForUsers`.
  Grace-period removal uses `setTimeout` keyed by `(userId, tabId)` so a fresh
  `hello` within the window cancels the removal and avoids a flicker.
- `apps/api/src/presence/subscription.ts` — `listPresenceSubscribers(userId)`
  unions friends (from `friendships`, which stores pairs as `(smaller, larger)`)
  and users who share any room (via `room_members`). Recomputed on every
  transition as the task requires; no caching.
- `apps/api/src/presence/sweeper.ts` — `setInterval` fires every
  `PRESENCE_SWEEP_INTERVAL_MS`, compares computed state against a
  `lastBroadcast` map, and publishes `presence.update` to each subscriber's
  `user:<id>` topic. Offline transitions evict the entry from `lastBroadcast`
  so we don't re-emit stale offlines. Broadcasts also reach the transitioning
  user's own tabs for multi-tab coherence.
- `apps/api/src/presence/ws-handlers.ts` — registers the `heartbeat` dispatcher
  handler and hooks `hello` / `close` on the WS lifecycle to register and
  grace-remove tabs.
- `apps/api/src/ws/lifecycle.ts` — new tiny event bus (`connectionLifecycle`)
  so presence can observe hello/close without touching the plugin's core
  message loop every time. Plugin changes are three lines: an import, an
  `emit('hello', conn)` after the tabId is attached, and an `emit('close', conn)`
  before the existing cleanup.
- `apps/api/src/presence/registry.spec.ts` — 10 deterministic tests with fake
  timers covering offline, online, afk, recovery, multi-tab, last-tab-removed,
  the grace-window cancel-on-reconnect, post-grace removal, snapshot defaults
  for unknown users, and the no-op path for an unknown tab.
- Wired imports added to `apps/api/src/routes/index.ts` so module-scope side
  effects (dispatcher registration, sweeper interval) run on boot.

## Key decisions

- **Lifecycle extension point instead of inlining presence calls.** Putting
  `connectionLifecycle.emit('hello', conn)` in the plugin keeps future
  features (e.g. typing indicators, metrics) from each wanting their own
  edit to plugin.ts. Three added lines in total, as the task budget allowed.
- **Transitioning user receives their own presence.update.** Not strictly
  required by the ADR but cheap, and it makes multi-tab rosters coherent
  without a separate "self-sync" path.
- **Sweeper keeps a `lastBroadcast` map.** Without it the registry's offline
  state is only observable at the moment of transition (the user is removed
  from the map as soon as the last tab goes), so the sweeper has to remember
  it was previously online/afk to emit the offline transition. Evicting on
  offline prevents the map from growing without bound.
- **Timers are `unref`'d.** Otherwise the grace `setTimeout` and the sweeper
  `setInterval` would keep Node alive during shutdown, causing
  `pnpm --filter api test` to hang if any presence code ever runs outside
  the explicit reset helpers.
- **No caching in subscription.** ADR-0002 cautions against premature caching
  and the task explicitly says "recompute each broadcast at MVP scale". The
  two queries are small; the cost to add a short-lived memo if it ever shows
  up in a profile is trivial.

## Cross-agent notes

- `getSnapshotForUsers` is exported but no one calls it yet. The task notes
  the route / WS-event wiring for snapshots is a later concern — the shape
  matches the task spec (`{userId, state}[]`), the shared `presenceSnapshot`
  envelope wraps it in `{entries: ...}` when the future consumer wires it up.
- `subscription.ts` queries `friendships` and `room_members` directly.
  Friends is still being built; when the friends agent's write path lands
  the query becomes meaningful for DM co-subscribers.
- Plugin edit is three lines and is the only non-presence file I touched in
  `apps/api/src/ws/*`. `connection-manager.ts` was not modified.

## Takeaways

- A tiny lifecycle event bus in `ws/` is a cleaner seam than sprinkling
  feature-specific calls through `plugin.ts`. Each new feature needing
  hello/close signals adds an `on('hello', fn)` at module scope and stays
  out of the hot path.
- Fake timers made the state-machine tests trivially deterministic — no
  real-wall sleeps, so the suite ran in 4 ms. Worth enforcing the pattern
  for any future timing-sensitive module.
- Evicting `lastBroadcast` on offline kept the sweep logic a one-liner
  `previous === next` check. Early drafts tried to track "was offline" as a
  distinct state and became knotted.
