# 2026-04-18 · Auth and sessions feature module

Entry covering the implementation of the auth and sessions HTTP surface on the
`worktree-agent-ada4f9c1` feature branch.

## What I built

Ten endpoints split across two route modules plus supporting helpers:

- `apps/api/src/auth/routes.ts` – register, sign-in, sign-out, me, password
  reset request + consume, password change, delete account.
- `apps/api/src/sessions/routes.ts` – list active sessions with `isCurrent`,
  revoke a single session by id.
- `apps/api/src/auth/password.ts` – argon2id hash + verify at OWASP 2026
  parameters (`m=65536, t=3, p=4`).
- `apps/api/src/auth/password-reset.ts` – 32-byte base64url tokens,
  SHA-256-hashed at rest, 30-minute TTL, single-use via `consumed_at`.
- Two spec files next to the helpers covering the pure logic: argon2 produces
  argon2id hashes with the right parameters, verify succeeds on the right
  plaintext, fails on wrong plaintext, survives malformed input; reset tokens
  hash deterministically with SHA-256 to 32 bytes.
- `apps/api/src/routes/index.ts` – uncommented imports for the two modules.

## Decisions

- **Delete-account as a single DB row delete in a transaction.** FR-AUTH-13
  says cascade owned rooms, remove memberships, null author on other
  messages. The schema already pins all the FKs to the right policy
  (`CASCADE` on owned rooms / memberships / bans, `SET NULL` on authored
  messages and uploaded attachments). Postgres performs the whole graph
  atomically, so the route handler is just a `DELETE FROM users WHERE id =
  $1` wrapped in `db.transaction(...)`. Cleaner than re-statements of the
  cascade in application code and matches the schema contract.
- **Bypassed the pre-existing `deleteSessionsForUserExcept` helper.** The
  helper in `apps/api/src/session/store.ts` uses `lt(sessions.id, keepId)`,
  which only deletes sessions whose UUID is numerically less than the
  current session id. That misses every session created after the current
  one. I didn't touch `store.ts` (out of feature boundary) and instead
  wrote a local `deleteAllSessionsForUserExcept` in `auth/routes.ts` using
  `ne(sessions.id, keepId)`. Coordinator: the shared helper needs fixing
  before other features rely on it – see "Proposals" below.
- **Reset link goes to the frontend origin.** Per the open-question default
  in `docs/requirements/01-auth.md`: `http://localhost:8080/reset?token=…`.
  Hard-coded; if we ever need an env-configurable origin that's a separate
  change.
- **Reset log goes via `req.log.info` with the `[AUTH]` prefix.** The spec
  says "log to stdout" – the Fastify logger is the project's stdout path.
  Format matches the exact string requested in the requirements.
- **No email enumeration anywhere.** Reset request returns 204 whether or
  not the email matched. Sign-in returns the same opaque `invalid_credentials`
  for unknown email and wrong password.
- **Username validation comes from `@agora/shared`.** The shared
  `usernameSchema` is `min(3).max(32)` with regex `/^[a-z][a-z0-9._-]*$/`.
  That is materially equivalent to the mission's
  `/^[a-z][a-z0-9._-]{2,31}$/` but framed differently (min/max length
  applied separately from the char-class regex). I stayed with the shared
  schema rather than re-expressing the constraint; the shared package is
  reference-only from this worktree.
- **Sign-in returns `userSelf` with `createdAt` as ISO string.** Matches the
  shared `userSelf` zod schema contract.
- **User-agent raw, clipped to 1024 chars.** `ua-parser-js` is mentioned in
  the hints but isn't in `package.json`; the requirements don't gate auth
  on parsed-UA labels (that's a sessions-view-presentation concern). Storing
  the raw header is correct and forward-compatible.
- **Password-change on wrong current password returns 401, not 400.** It's
  a credential check, not an input-validation failure. Symmetric with sign-in.

## Pitfalls

- **`exactOptionalPropertyTypes: true` meant I had to be careful never to
  pass `undefined` into Drizzle `.values(...)`.** I only insert the
  non-optional columns on `users` and let defaults fill `createdAt` /
  `updatedAt`. `deletedAt` is left null by absence.
- **Module augmentation for `FastifyRequest` lives in
  `apps/api/src/session/plugin.ts`.** As long as the plugin module is
  imported transitively by `server.ts` (which it is), my route handlers see
  the augmented `req.session` / `req.user` types. No duplicate declarations
  needed.
- **`deleteSession` in the store scopes by `(id, userId)`.** Perfect for
  `DELETE /api/sessions/:id` – a user can only revoke their own sessions;
  unknown id falls through to a 404.

## Proposals for the coordinator

1. **Fix `deleteSessionsForUserExcept` in `apps/api/src/session/store.ts`.**
   Replace `lt(sessions.id, keepId)` with `ne(sessions.id, keepId)`. The
   current implementation only kills half the sessions (the ones with
   lexicographically smaller UUIDs). Once fixed, my local wrapper in
   `auth/routes.ts` can be deleted and the routes can call the shared
   helper directly.
2. **Consider an `errorCode` convention in `@agora/shared`.** I'm returning
   `{ error: 'email_taken', … }` / `{ error: 'username_taken', … }` / etc.
   If we're going to test these codes from the web app it'll want a shared
   enum / union rather than string literals sprinkled around.
3. **Username regex in shared vs requirements doc.** The shared
   `usernameSchema` uses `min(3).max(32)` + a regex without repeat bounds;
   the requirements and the mission brief both show the fused
   `/^[a-z][a-z0-9._-]{2,31}$/`. They're equivalent, but a single
   expression would make the constraint easier to audit. Non-blocking.
4. **Rate-limit on `/api/auth/sign-in`.** The requirement hints at 10/min
   per IP+email. `@fastify/rate-limit` is installed but I didn't wire it
   to keep this commit focused on the route surface. Worth a follow-up
   before the smoke run.

## Takeaways

- The schema carries enough information that account-deletion was two
  lines of route code. Letting FKs do the work beats re-stating cascade
  rules in application code.
- The route-module registry pattern is clean: `addRouteModule` at module
  scope, a single import in `routes/index.ts`, and the feature is plugged
  in without touching `server.ts`.
- Worktree isolation is real: no `node_modules`, no ability to invoke the
  workspace toolchain from inside it. For the next worktree agent, the
  coordinator should consider seeding a symlink farm to the parent's
  `node_modules` so typecheck/test can actually run locally.
