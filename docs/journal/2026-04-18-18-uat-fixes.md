# 2026-04-18 · UAT fixes — inline edit, modal delete, image previews, flex chain

Follow-on to `2026-04-18-17-spec-prod-sweep.md`. After the spec-vs-prod sweep closed out, I did a real end-to-end UAT in Chrome and the user did the same. Four findings — one of them a sleeping-giant layout bug that had been hidden all along.

## What the user spotted

1. **Ugly edit message prompt** — `window.prompt('Edit message', msg.body)` inherited from wave-3.
2. **Pasted an image but only the stripe placeholder shows** — `FileCard` was rendering `[image preview]` (a DS mockup placeholder) instead of the actual image.
3. **Delete confirmation is also ugly** — `window.confirm('Delete this message?')`, same wave-3 debt.
4. **Edit/delete flicker on hover** — the actions were conditionally rendered on `hovered`; every mouseenter/leave remounted them and shifted row layout.

All four were explicitly called out as deferred follow-ups in the design-audit journal ("window.prompt/confirm replaced with inline editing and the Modal primitive" was tagged as a >10-min follow-up). UAT made them all priority.

## The bonus: `minHeight: 100vh` was killing the flex chain

While verifying the UAT fixes I re-ran the Playwright `large-history` e2e. It timed out waiting for "seeded message 10000" to be visible. Probing the DOM manually, the message *was* rendered — but `scrollHeight == clientHeight == 101248px` on the scroller. In other words, the scroller had expanded to the full height of all its content; the flex-1 + overflow:auto chain MessageList relied on wasn't constraining it.

Root cause: `RootLayout`'s outer used `minHeight: 100vh`. At first glance that's "at least the viewport tall", which sounds right. But under flex-1 children with overflow:auto, the parent must be *exactly* viewport-tall for the `flex: 1` descendants to cap their own height and scroll internally. With `minHeight`, the parent grew with content, and the IntersectionObserver at the top of MessageList kept firing because the top sentinel was always visible inside an unbounded scroller. Result: every fetchNextPage cascaded, the full 10k history loaded into memory, and the virtualiser rendered 3650+ items in one column.

This bug had been there since day one. The symptom was hidden because:

- Normal browsing never hit it — `overflowY: auto` on the MessageList did *partially* work, but only because the viewport itself provided an implicit height constraint (body scrolls). On a fresh page load with 50 messages the virtualiser fit in the visible area so no over-fetch.
- The e2e `getByText` matched any element in the document, including ones below the fold. It passed because the 10000th message was *somewhere* in the DOM — never mind that 9950 others had also loaded.

The change in this pass — MessageActions going always-mounted — is what finally broke the test: the always-mounted edit/delete text entered the body span's `textContent`, so `getByText('seeded message 10000', { exact: true })` no longer matched. That forced a proper investigation and the real bug fell out.

One-line fix: `height: '100vh'` on the RootLayout outer.

## Each UAT fix

### Edit — inline editor

Matches the classic-chat convention (Slack, Discord, IRC clients) and fits the DS "dense, text-first" aesthetic better than a modal would. In-row textarea replaces the body text, with:

- Focus on mount, cursor at end of existing text
- `⌘⏎` or `Ctrl+⏎` to save, `Esc` to cancel
- Save button gated on "non-empty and different from the original"
- Persistent "⌘⏎ to save · Esc to cancel" hint

Implementation: new `InlineEditor` component rendered inside `MessageRow`'s children slot when `editing === true`. State (`editingId: string | null`) lives on `MessageList`.

### Delete — Modal confirmation

Claude Design's `Screens.html` has a `RevokeConfirmModal` for session revocation with the exact pattern we needed: title + body explaining what happens + monospace preview of the affected entity + Cancel/Danger buttons. `DeleteConfirm` mirrors that directly:

```tsx
<ModalScrim onClose={onClose}>
  <Modal title="Delete message" width={440} onClose={onClose}>
    <Col gap={12}>
      <div>…explanation…</div>
      <div style={{ background: paper1, padding: '8px 12px', fontFamily: mono }}>
        {msg.body || '(attachment)'}
      </div>
      <Row gap={8} style={{ justifyContent: 'flex-end' }}>
        <Button onClick={onClose}>Cancel</Button>
        <Button variant="danger" onClick={onConfirm}>Delete</Button>
      </Row>
    </Col>
  </Modal>
</ModalScrim>
```

State (`confirmDelete: MessageView | null`) lives on `MessageList`. Esc closes through the Modal's existing key handler; click-outside closes via `ModalScrim`.

### Image previews — actually render images

`FileCard` was rendering a mockup-era placeholder div saying `[image preview]` with a stripe background. Now renders `<img src={href}>` (the `/api/attachments/:id/download` path; cookie auth carries via the browser) with the stripe kept as a CSS background so there's shape during load or 404. `target="_blank"` so the anchor opens the full-size view. The uploader's `comment` now shows in the metadata line too (it was only plumbed on the non-image branch before).

### Flicker — always-mount + opacity transition

Was:

```tsx
{hovered ? <MessageActions msg={msg} .../> : null}
```

That remounted the actions every time the cursor crossed a row boundary. Combined with the layout shift (conditional markup ≠ same height), the transition was visibly janky. Now:

```tsx
<MessageActions visible={hovered} canEdit={...} canDelete={...} .../>
```

…with

```ts
style={{
  opacity: visible ? 1 : 0,
  pointerEvents: visible ? 'auto' : 'none',
  transition: 'opacity 80ms ease',
}}
```

No remount, no layout shift, no flicker.

**Side-effect that caught the scroll bug**: with MessageActions always in the DOM, its "edit delete" text now contributes to the body span's `textContent`. Playwright's `.getByText('seeded message 10000', { exact: true })` needs an exact match on a node's text content — so it stopped matching. Wrapping the message body in its own `<span data-testid="message-body">` gives the test a clean target and keeps accessibility labelling tidy.

## Takeaways

- **UAT finds what neither audits nor tests can.** All four findings were things a human with a mouse noticed in seconds; none of them surfaced in three waves of internal audits, one Claude Design audit, and a spec-vs-prod production sweep. The code-level review tools don't render the page, and e2e tests don't form opinions about aesthetics. A three-minute click-through remains a mandatory step.
- **One UAT fix can unmask an older bug.** The MessageActions always-mounted change polluted `textContent` for the exact-match test, which forced me to investigate, which revealed the `minHeight: 100vh` issue that had been there since the very first commit. If I'd just "fixed the test" to match non-exactly, the sleeping giant would have stayed asleep until demo day when a reviewer scrolled to an older message in a busy room. Test failures are often louder messengers than we credit them for.
- **`minHeight` vs `height` on flex roots is exactly this bug.** Worth writing down because it's a genuinely counter-intuitive piece of CSS: `minHeight: 100vh` sounds like the safer option ("at least this tall"), but breaks overflow-based scrolling in any flex-1 descendant. For a layout that contains a virtualised scroll region, `height: 100vh` is the invariant.
- **Inline edit beats modal edit for text-first chat.** Composing a new message and editing an existing one are the same activity; the UI should treat them the same way. Popping a modal for edit is what messaging apps built on top of DB-record abstractions do (one record, one modal). Chat is a river, and the river is the UI.
- **Audit-item triage has a known failure mode: "real feature, not pure alignment" punting.** I used that framing to defer inline edit, modal delete, toast host, DM freeze, and composer attach after the design-audit pass. All four eventually shipped in the following follow-up commit. Next time, land alignment items together with the features they naturally couple to, rather than splitting by effort-class.
