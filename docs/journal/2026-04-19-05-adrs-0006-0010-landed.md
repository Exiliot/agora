# 2026-04-19 · ADRs 0006-0010 all accepted and landed

The five ADRs proposed after the parallel audit pass are all in now. Sequential dispatch, one subagent per ADR (plus one paired dispatch for the two that both touched `config.ts`), each ending with `Status: Proposed` → `Status: Accepted` and an Implementation section carrying the landing SHA.

Parallelisation was considered and rejected. The five ADRs overlap on four hot files (`apps/api/src/config.ts` for 0007 and 0010, `packages/shared/src/ws/events.ts` for 0006 and 0009, `apps/web/src/app/WsProvider.tsx` for 0009 plus any later), so a parallel fan-out would have produced merge conflicts. Sequential was lower risk and only marginally slower – four dispatches instead of five, with 0007+0010 batched.

## What this closes

- **Security**: the 2026-04-19 audit's H1 (reset link logged unconditionally) and H2 (cross-site WebSocket hijack) are both closed. The only audit-level H3 remaining is the XMPP internal routes, which is gated by `ENABLE_XMPP_BRIDGE=1` and follows ADR-0005; that's a separate sidecar and out of scope here.
- **Product integrity**: the "nine dropped events" critical is closed. Moderation and friendship changes now propagate live without reload; the adjacent privacy leak (banned user kept receiving room messages) closes too because the server unsubscribes their WS from the room topic as part of the remove/ban/delete transaction.
- **Performance**: `message.send` retry-under-flaky-network no longer produces a double row. The LRU + partial unique index is a belt-and-braces pair – the LRU covers the happy path, the index covers any race the LRU can't (process restart, multi-replica).
- **Accessibility**: the overlay contract closes H1 (heading hierarchy on auth routes) and H2 (`NotificationMenu` focus contract). The Modal/popover primitives now all flow through the `useOverlay` hook so future overlays inherit the contract for free.

What's *not* closed here: the notifications feed N+1 (perf H2), the `mark.read` never-fires bug (product critical #1), the client's missing UI for `DELETE /api/users/me` (product critical #2), and the `--mention-fg` / `--mention-bg` contrast failure (a11y H3). Those are follow-ups.

## Process notes

- The subagent-per-ADR shape was the right call. Each agent got a fresh context, the ADR spec verbatim, a concrete set of files to read, a concrete set of files to change, and the verification steps. No agent needed to ask clarifying questions during implementation.
- Status-flip is part of the agent's job. Every dispatch instruction included the Proposed → Accepted edit and the Implementation section append. That keeps ADR hygiene automatic rather than a separate later sweep.
- Two-commit pattern per ADR (feat + docs SHA backfill) is clumsy but honest. The rule "never amend" is the right rule; the extra commit is its cost.

## Takeaways

- Audits → proposed ADRs → sequential subagent implementation is a clean loop. Four specialist audits found 94 observations; five ADRs captured the cross-cutting themes; five (batched to four) implementations closed them. No agent has to hold the full graph in its head because the ADR is the contract.
- The one place this pattern didn't quite fit was 0009's "compile-time exhaustive switch". The agent needed to actually verify the switch catches missing events – which meant temporarily breaking the shared union, running tsc, reading the error, and reverting. That's not something a pure implementation agent usually does. Flagging it here as a pattern: when you claim a compile-time invariant, prove it with a break-and-revert.
