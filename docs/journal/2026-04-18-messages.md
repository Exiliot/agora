# 2026-04-18 ¬∑ Messaging feature module

Implementation of the messaging surface: real-time send/edit/delete over
WebSocket, HTTP history and backfill, DM opening, and the sidebar conversation
list. Built in an isolated worktree alongside the rooms and friends agents.

## Scope delivered

- `apps/api/src/messages/permissions.ts` ‚Äî `canAccessRoom`, `canSendDm`,
  `loadDmForUser`, `canAccessDm`. All query the DB on every call (AC-6 from
  CLAUDE.md: never cache auth decisions).
- `apps/api/src/messages/unread.ts` ‚Äî `listOtherParticipants`,
  `incrementUnread`, `resetUnread`, `getUnread`. Accepts either `db` or a
  transaction handle derived from `Parameters<Parameters<typeof db.transaction>[0]>[0]`
  so counters participate in the send transaction.
- `apps/api/src/messages/history.ts` ‚Äî `fetchHistory` for before/since cursor
  pagination, `hydrateMessages` joining authors + attachments, `hydrateMessage`
  for single-row hydration.
- `apps/api/src/messages/ws-handlers.ts` ‚Äî registers `message.send`,
  `message.edit`, `message.delete`, `mark.read` on the WS dispatcher at
  module scope. Publishes `message.new`, `message.updated`, `message.deleted`
  on the conversation topic; `unread.updated` on the per-user topic.
- `apps/api/src/messages/routes.ts` ‚Äî `GET /api/conversations/:type/:id/messages`
  (handles both `before` and `since` cursors), `POST /api/dm/open`,
  `GET /api/conversations`. Each route gates on `requireAuth` via `preHandler`.
- Tests in `apps/api/tests/integration/messages.spec.ts` covering permission
  logic and bus scoping.

## Key decisions

- **Cursor semantics.** `before` and `since` are mutually exclusive; returning
  a 400 if both are supplied keeps the client contract crisp. Default limits
  are 50 (history) and 200 (since); maxes 100 and 500.
- **UUIDv7 everywhere.** New message ids come from `uuidv7()` ‚Äî monotonic per
  process, so comparing by `id` is equivalent to `(created_at, id)` for
  pagination. Saves a composite index scan.
- **Message-send transaction.** Insert + unread increments for each recipient
  run in a single `db.transaction`. Bus publish happens after commit so
  subscribers never see a message without the unread counter being updated.
- **Soft delete.** `message.delete` sets `deletedAt` and clears `body`. The
  attachment rows are left alone in this pass ‚Äî FR-ATT-7 says bytes persist
  and there's no attachment feature wired yet to test the detach path.
- **Reply target validation.** If `replyToId` is supplied, we verify the
  target message exists in the same conversation before inserting. Spec says
  the referenced message must exist at send time.
- **Unread fan-out on mark.read.** Only the caller gets `unread.updated` with
  count 0 (on `user:<callerId>`). Multi-tab sync falls out of this because
  every tab of the caller is subscribed to that topic.
- **DM idempotency.** `POST /api/dm/open` reads existing ‚Üí inserts with
  `onConflictDoNothing` ‚Üí reads again. The second read is necessary because
  the conflicting insert doesn't return a row.

## Cross-agent notes

- DM permission checks query `friendships` and `user_bans` directly. The
  Friends feature is being built by another agent in parallel; the tables
  exist in the schema so the queries compile, but real end-to-end DM flows
  depend on that agent's write paths.
- `attachmentIds` on `message.send` is accepted but not yet wired to the
  `attachments` table updates. The attachments feature is a separate
  agent's territory; when it lands, a follow-up change will link uploaded
  attachment rows to the inserted message inside the transaction.
- The WS ack envelope shape in `apps/api/src/ws/plugin.ts` uses
  `{type: 'ack', payload: {reqId, result}}` while the shared `ackEnvelope`
  schema is `{type: 'ack', reqId, result}`. I mirrored the plugin's existing
  shape to stay consistent with the built-in handlers (`hello`,
  `subscribe`, `unsubscribe`). If the wire shape is standardised later, all
  four sites update together.

## Takeaways

- Writing permission checks as small typed helpers rather than inline gates
  paid off immediately ‚Äî the same three functions serve both HTTP routes
  and WS handlers with zero duplication.
- Drizzle's transaction type is awkward to reconstruct manually; extracting
  it via `Parameters<Parameters<typeof db.transaction>[0]>[0]` is uglier to
  read but doesn't drift when drizzle bumps.
- The bus is trivially testable ‚Äî subscribe, publish, assert ‚Äî which made
  the topic-scoping test a one-liner. Good sign for the runtime behaviour.
