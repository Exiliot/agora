# 2026-04-18 · Scroll-history UX — anchors + jump-to-latest

Follow-on to `2026-04-18-18-uat-fixes.md`. User asked: "in a chat with virtualisation, scrolling up is infinite with no expectations about when it ends — how can that be sorted better?"

Real UX gap. The virtualiser is correct but the surface offered no where-am-I signals: no day separators, no channel-start anchor, and no way back to the present once you'd scrolled away.

## Options in rough value-per-effort

| Idea | Value | Cost | Shipping? |
|---|---|---|---|
| Day separators between date boundaries | Highest — the anchor every long-history chat has | S | Yes |
| Jump-to-latest pill when scrolled up | High — standard in every modern chat; saves endless scroll-back | S | Yes |
| Richer "start of conversation" marker (date, description) | Medium — turns the terminal state into a proper signal rather than silence | XS | Yes |
| Visible loading spinner at the top | Medium — current "loading older…" is 11px ink-3, easy to miss | XS | Yes |
| Scroll-position counter ("3,420 of 10,000") | Accuracy at the cost of feeling like a file viewer, not a chat | S | No — aesthetic mismatch |
| Unread-since-last-visit divider | High long-term, but couples to backend unread tracking | M | Deferred to a future pass |

## What shipped

### Day separators

Rendered inside each virtual row, guarded by a "date differs from previous message" boundary check. Hairline rules either side of a mono label:

```
──────────  Today  ──────────
[21:20] alice: hey
[21:20] alice: are you around
──────────  Yesterday  ──────────
[18:00] bob: on my way
```

Label rules:

- Same day as today → `Today`
- Day before today → `Yesterday`
- Anything else → `Mon 14 Apr 2026` (weekday · day · short-month · year, via `toLocaleDateString`)

`role="separator"` with `aria-label` for screen readers. Because the separator renders inside a virtual row, scroll position stays accurate as pages page in above — the virtualiser's `measureElement` picks up the extra height.

### Start-of-conversation marker

Was:

```
— beginning of conversation —
```

Now:

```
─────────  start of conversation  ─────────
     first message Mon 14 Apr 2026
```

Two-line block: rules flanking "start of conversation" label + a secondary line with the `formatDateLabel` of the oldest message. Matches the DS hairline-rule idiom.

### Top loader

Was 11 px ink-3 text that read "loading older…" — easy to miss and didn't look like a loading state. Now:

- 12 px ink-2 mono
- A 10×10 border-spinner next to the text
- Reads "loading older messages…" instead of "loading older…"

Keyframe (`agora-spin` in the components layer of `base.css`) is shared — any future spinner can reuse it. Calm 700 ms rotation tuned to feel classic rather than racing.

### Jump-to-latest pill

Small teal mono pill at bottom-right of the scroller:

```
↓ jump to latest
```

Shows when the scroller is more than 48 px from the bottom. Click smooth-scrolls to `scrollHeight`. Wrapper around the scroller gets `position: relative` so the pill anchors to it rather than to the page.

### The `useEffect` listener that wasn't firing

Initial wiring used `useEffect` + `addEventListener('scroll', …)` on `scrollRef.current`. Simple pattern, should have worked — didn't. Debug confirmed the listener was registered (effect ran, no errors) but `scrollTop` changes from user-scroll never propagated to `isAtBottom` state; the pill never appeared. No clean theory — likely a subtle interaction with the virtualiser's own ref + the wrapper refactor that happened around the same time (fresh rendered scroller replacing the one the effect had captured).

Swapped to React's `onScroll` prop bound inline. Fires cleanly, no remount sensitivity, three fewer lines of code. Kept a comment explaining *why* so a future refactor doesn't "clean up" back to the effect-based version.

## Deferred

**Unread-since-last-visit divider.** Classic pattern (Slack's "New"/Discord's "New messages since…"). Needs a server-side notion of "the message id at which the user last left this conversation" — the schema already has `last_read_message_id` per user per conversation (for unread counters), so the data is there. Rendering is a separator row at that boundary. Real work is the UI: it should appear on first render but not vanish on subsequent scrolls, and clicking away shouldn't re-anchor it to the new bottom. Planned for a future pass.

## Verification

- `pnpm typecheck` clean.
- `npx playwright test` 7/7 (large-history now also renders day separators cleanly in its scroll through 95+ progressive pages).
- Manual UAT: scroll up from the bottom of a 10k-history, jump-to-latest pill appears immediately, click it, smooth scroll to bottom, pill disappears. Day separators render correctly between messages that span day boundaries (tested by seeding 20 messages at `now - 1d`, 20 at now).
- `pnpm lint:tokens` clean.

## Takeaways

- **Virtualisation without anchors feels like a wall of text.** Date separators are a near-free fix (one boundary check, one span render) that transforms the scroll experience. Should have been in from the start — noting that for the next chat UI.
- **"Jump to latest" earns its complexity budget.** In any chat with more than a screen of history, it's the single most clicked non-message affordance. Not a nice-to-have.
- **`useEffect` + `addEventListener` on a ref is a minor footgun.** When the thing you want to listen on is a DOM node React already owns, `onScroll` / `onKeyDown` / etc. are the idiomatic choice — they're bound to the actual mounted element every render and don't compete with the ref-capture order. I've written the effect-style version dozens of times without trouble, which is exactly why the failure was disorienting; debugging took longer than the actual fix.
- **Small anchors beat progress indicators for chat.** A "3,420 of 10,000" counter was tempting — precise and unambiguous. But it reads as "I am in a list of records", which is the exact frame classic web chat is trying to avoid. "Today / Yesterday / Mon 14 Apr" is how people mentally index conversation time; let the UI mirror that.
