Follow-on to `2026-04-18-wave-1.md`. Wave-1 took the sev-critical and most sev-high findings in one bundled commit. Wave-2 is the smaller, more considered pass: items that were real refactors or touched multiple files, where bundling them with wave-1 would have made review harder.

## What shipped

### Typed WS event handlers (audit code-quality #5)

The WS plugin already validated every incoming event against `clientToServerEvent`, a zod discriminated union in `@agora/shared`. But once the event reached the feature handler via `dispatchWsEvent`, the type got flattened back to `ClientToServerEvent` and each handler cast its payload via `as`:

```ts
// Before
registerWsHandler('message.send', async (ctx, evt) => {
  const { reqId, payload } = evt as MessageSendEvent;
  ...
});
```

That's fragile: a future addition to `clientToServerEvent` whose discriminator matches by accident would silently pass through the wrong handler and corrupt a DB insert. Fix was to thread the union through the dispatcher so handlers receive the narrowed variant without casting:

```ts
// After
type EventOf<T extends EventType> = Extract<ClientToServerEvent, { type: T }>;
type HandlerFor<T extends EventType> = (
  ctx: WsContext,
  event: EventOf<T>,
) => Promise<void> | void;

export const registerWsHandler = <T extends EventType>(
  type: T,
  handler: HandlerFor<T>,
): void => { ... };
```

Handler bodies drop the cast:

```ts
registerWsHandler('message.send', async (ctx, event) => {
  const { reqId, payload } = event;  // already typed as MessageSendEvent
  ...
});
```

Net: ~20 lines of dispatcher types, ~50 lines of `as …` casts deleted. Type regressions on `message.send` / `message.edit` / `message.delete` / `mark.read` are now compile errors instead of runtime surprises.

### `@fastify/helmet` + CSP (audit security #9)

The api HTTP surface returned zero security headers. Registered `@fastify/helmet` with `contentSecurityPolicy: false` because:

- The web is served through its own nginx, which is where CSP actually belongs (route-specific, different policies for the SPA vs API vs attachments).
- Helmet's default CSP would fight React's inline style hashing in development, adding a nonce/hash pipeline that's not worth it for a hackathon scope.

So helmet handles the cheap wins — `X-Content-Type-Options: nosniff`, `X-DNS-Prefetch-Control: off`, `Strict-Transport-Security` (noop on http-only localhost, correct once TLS is in front), `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer` — and nginx carries the CSP:

```nginx
add_header Content-Security-Policy "default-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data: blob:; connect-src 'self'; frame-ancestors 'none'; form-action 'self'" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "no-referrer" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
```

`'unsafe-inline'` on `style-src` only — React sets inline `style=` attributes on rendered components; the design-system primitives use them heavily. Scripts are `'self'` only; no eval, no inline scripts, no CDN.

### `trustProxy: 1` (audit security, implied)

The rate-limiter from wave-1 keyed on `req.ip`. Without `trustProxy`, every request coming through the web container's nginx showed up with the same IP (the nginx container's internal address), defeating the rate-limit entirely. Setting `trustProxy: 1` tells fastify to honour the single upstream `X-Forwarded-For` hop but not chains, which is the correct semantic for a single reverse-proxy topology — chaining would let clients spoof.

### Unified error response shape (audit code-quality #12)

Auth routes returned `{error: 'invalid_input', issues}`; rooms/messages/friends returned `{error: 'validation', message}`; some 409s were `{error, code, message}`. The web `ApiError` tolerated any shape via optional-chaining but the translated user-facing message was unpredictable.

Added a shared `setErrorHandler` in `server.ts`:

```ts
app.setErrorHandler((err, req, reply) => {
  const narrowed = err as { statusCode?: number; code?: string; message?: string };
  const status = narrowed.statusCode ?? 500;
  if (status === 500) req.log.error({ err }, 'unhandled');
  return reply.code(status).send({
    error: narrowed.code ?? (status >= 500 ? 'internal' : 'error'),
    message: status >= 500 ? undefined : narrowed.message,
  });
});
```

Validation paths still return `{error: 'validation', issues: [...]}` at 400 with the zod issue list — that shape carries extra structure the client uses for field-level messaging. Legacy route-level `reply.code(...).send({error, code, message})` bodies still work unchanged; the handler only catches thrown errors.

## Takeaways

- **Typed dispatchers pay for themselves in one refactor.** The dispatcher-type change itself was ~15 minutes; dropping all the `as` casts cleaned up ~50 lines and the next refactor that adds a new WS event type will catch misses at compile time instead of in a production log. Worth the cost of entry on any project that dispatches on a discriminated union at scale.
- **Who owns CSP is a real design question.** Registering helmet everywhere is easy; deciding that nginx carries the CSP while helmet carries the single-line headers is the judgement call. Two authors of CSP + api-and-web not aligning would have produced contradictory `Content-Security-Policy` headers. Worth writing down so a future contributor doesn't add helmet's CSP in a reflex.
- **Trust-proxy is a silent correctness bug.** Rate limiting appeared to work; checking against a burst of requests showed every hit resolving to the same "ip" (the nginx container), so either every user was one bucket or the bucket was infinite depending on config. Noticed only because the wave-1 rate-limit tests were run against a real deploy shape (compose, not dev-server). Cheap to fix, easy to miss.
