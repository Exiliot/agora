## Kickoff (11:00 UTC, 2026-04-18)

Task dropped at 11:00 UTC: a classic web chat server. Rooms (public/private), DMs, friends/contacts, multi-tab presence (online/AFK/offline with 1-minute idle rule), active sessions with individual logout, file/image sharing with ACLs tied to room membership, message replies/edits/deletes, infinite-scroll history (10k+ messages per room), 300 simultaneous users, 1000-member rooms. Optional Jabber/XMPP federation with a load test as the stretch goal.

A much larger surface than Denis implied on Thursday. "Doable in a day" is a stretch. Two days is workable with disciplined scoping.

Two hard architectural constraints came out of the kickoff Q&A, both of which I'm treating as forced choices rather than free decisions:

1. **Sessions server-side, in DB, not JWT.** Denis was explicit: the spec requires immediate revocation (view active sessions, log out one) and user-to-user bans must take effect *now*. JWT fights this for the rest of the event.
2. **In-memory message queue + in-memory presence tracking.** Naive DB read/write on every message/heartbeat doesn't scale to 300 concurrent users. Denis said this twice on the kickoff call.

## Decisions taken in this session

Before writing a line of application code:

- **Repo name**: `agora` — Greek public gathering place. Short, classical, apt for a chat. Public repo at `github.com/Exiliot/agora`, created empty; we work locally and push only when the scaffolding is sound.
- **Stack**: Node 24 + TypeScript + Fastify 5 + Postgres 16 + React 19 + Vite. pnpm workspaces as the monorepo glue. Drizzle ORM + drizzle-kit migrations. `@fastify/session` with a DB-backed store. Argon2id for password hashing. `@fastify/websocket` for realtime. In-process EventEmitter for pub/sub — no Redis (unnecessary for 300 users on a single node, and adds compose complexity).
- **Frontend**: Vite + React 19 + TanStack Query + Zustand + Tailwind. Design tokens ported verbatim from Claude Design. No UI kit — custom primitives match the aesthetic (warm paper, oxidized teal, IBM Plex Mono chat, hairlines over shadows, no bubbles).
- **Testing**: Vitest for units/integration, Playwright for end-to-end. Smoke test runs in the docker-compose stack before every push.
- **Linting/formatting**: Biome, single tool. Already installed globally via Volta.
- **Conventional commits without tooling.** No husky, no commitlint. Discipline + the format (`type(scope): imperative message`) is enough for a two-day event.
- **Jabber/XMPP federation**: deferred to Phase 2. Sidecar design (Prosody container subscribing to our internal bus) means no breaking changes to the main app when picked up later. Captured as an ADR so the escape hatch is explicit.
- **Design**: 90-minute Claude Design session with Opus 4.7 produced a complete design system — tokens, primitives, components, composed screens — imported wholesale into `docs/design/`. The aesthetic is *"classic web chat, IRC lineage, rocket.chat era — not a modern messenger."* Six principles enforced: text first, density honestly, no bubbles, one accent one hue, hairlines over shadows, mono for truth.

## Philosophy codified

Two meta-rules I'm writing into CLAUDE.md to stop the agent drifting:

1. **Context before code.** Every feature starts by reading (or writing) a requirements file, then a plan file, then implementation. No speculative refactoring.
2. **The delivery contract is a check, not a wish.** `git clone && docker compose up` must work on a clean machine. Denis said ~50% of past submissions failed this single check. We check it every push, not just at the end.

### The one bug caught by `/smoke`

The web Dockerfile's `pnpm build` step failed with `TS6310: Referenced project ... may not disable emit`. The culprit was a composite `tsconfig.node.json` with `noEmit: true`, which is contradictory. Removing the project reference (vite handles its own config file) fixed it. This is exactly why `superpowers:verification-before-completion` and the delivery-contract check exist — without running the actual build, the broken tsconfig would have shipped.

## Takeaways

- The pre-event work paid for itself in the first 30 minutes: zero tool-setup scramble, clear tech decisions ready to validate against the actual brief.
- The brief is bigger than advertised. Scope discipline matters more than raw velocity — pick the smallest end-to-end slice that passes the delivery contract and build outward.
- Claude Design produced a much more opinionated, coherent aesthetic than I'd have invented from scratch in the same time. Worth the 90 minutes, especially because the exported tokens drop straight into CSS custom properties.
- The architectural forcing functions (server-side sessions, in-memory queue, in-memory presence) are not in the PDF — they come from the kickoff Q&A. Reading only the PDF would send you down a wrong path for each of them. The written spec is never the full spec.
- **Documentation-first pays off even in a two-day event.** By the time the first line of application code was written, the stack choices were well-justified, the data model was settled, the WS protocol was drafted, and the ADLC machinery was in place. That's hours-per-feature saved over the next day.
- **Always run the delivery contract at the end of the foundation phase.** Catching the tsconfig bug saved a Monday-morning panic.
- **Conventional commits without tooling works.** Eighteen commits, every one reading like a human wrote it, all in British English, no em dashes, no AI attribution. Discipline beats husky for a two-day event.
