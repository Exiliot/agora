# 2026-04-18 · Wave-1 audit closure

The afternoon session kicked off four parallel audit agents (security, performance, a11y, code quality) against the then-current `main`. This entry captures the triage and the first round of fixes — the ones that land before anything else because they'd fail a review on sight. Two commits stapled together here: `7654a6` (mid-triage critical fixes, done *while* the audit agents were still finishing) and `128a71` (the wave-1 fix bundle proper).

## Triage rules

1. Critical findings land first, sev-high next, everything else parked to a later wave.
2. If a finding is both trivial and load-bearing (ex: a missing DB index on a per-request query), fix it immediately rather than batch.
3. One wave = one landable commit. Don't let a 50-file cleanup commingle with a 2-file security fix.
4. Every fix writes a short line into the table below so the audit → fix mapping survives the session.

## Wave-1 proper (shipped in `128a71`)

### Security

- **`@fastify/rate-limit`** is in `package.json` but was never registered. A new `auth/rate-limit.ts` registers it globally-off, then per-route: 10/min on `/api/auth/*` keyed by IP for anonymous routes, keyed by session id for `password-change`.
- **WS origin check**. The upgrade handler now validates `req.headers.origin` against an allow-list and closes 4403 on mismatch. Without it the session cookie rides along on any cross-origin WS connection (classic CSWSH); combined with the fixed subscribe ACL it's now a single, consistent gate.
- **No committed `SESSION_SECRET`**. The default in `config.ts` was cosmetically used (cookies are not signed today) but fails review every time. `SESSION_SECRET` became optional; when unset, the api generates a 48-byte random secret at boot and warns that sessions won't survive a restart. `docker-compose.yml` no longer sets the env var, so a fresh deploy gets a fresh secret. Anyone wanting session-survival across restarts sets the env var themselves.

### Performance

- **Per-recipient `incrementUnread` was serial inside the `message.send` transaction** — 1000 round-trips in a 1000-member room. `incrementUnreadForMany` is a single `INSERT ... ON CONFLICT DO UPDATE` batched across all recipients; one round-trip regardless of room size.
- **`bus.publish` stringified per subscriber**. For a message going to hundreds of WS sockets this is hundreds of `JSON.stringify` calls over the same payload. `bus.publish` now pre-stringifies once when `subs > 1` and passes the serialised string through to handlers; `conn.sendRaw(serialised)` skips re-serialisation. (Side-effect: the existing publisher-integration specs went from passing-by-accident to passing-by-extra-`undefined`-arg, which we didn't notice until wave-3 — see that journal.)

## What got deferred to later waves (and why)

Not every audit finding got fixed in wave-1 — some because they're real refactors masquerading as fixes, some because they're user-visible and better done together with a UI change, some because the wave-1 commit was already 300 lines.

- **Typed WS handlers** (audit code-quality #5: `as …` casts in `ws-handlers.ts`). Real fix is to thread the discriminated union through the dispatcher — ~30 lines of type plumbing, merits its own commit. Done in wave-2.
- **`@fastify/helmet` + CSP**. Deferred to wave-2 so it lands with the unified error-shape work; both are "server hardening".
- **Unified HTTP error shape** (audit code-quality #12). Requires touching every route module; easier after wave-1's security-critical fixes are in.
- **`attachmentIds` linking** (audit security #42, code-quality #6). Deferred to wave-3 — the composer upload path was already half-working; pairing the fix with a once-over of the attachment flow made more sense than doing it in isolation.
- **Event-publisher shape drift vs shared schemas** (code-quality #7). Not user-visible today; bundled into wave-3 with the attachmentIds work.
- **DB-backed integration tests**. Half-day of fixture work plus CI changes; the MVP is shipping without them. Noted as debt, not a wave-1 fix.

## Takeaways

- **Audit agents are most valuable in parallel, not in series.** Running four agents at the same time meant the fixes could be triaged as a batch — prioritising a sev-critical WS ACL bug against a sev-high missing DB index is easier when both reports arrived within a minute of each other. Sequential audits tempt you to fix each one as it arrives.
- **The "mid-triage critical commit" pattern is worth doing on purpose.** Waiting for all four reports before starting fixes costs hours. Fixing the sev-criticals the moment an agent flags them, then doing a "wave-1 proper" commit for the bundled high-severity fixes, compresses the timeline dramatically.
- **Every wave-1 item was cheap.** Five criticals, five highs for security, two highs for performance, five for a11y — ~500 lines of changes across 16 files in under 90 minutes once the reports were in. The expensive ones (typed WS, unified errors, attachment linking) were correctly deferred. That's the healthy shape: waves don't get bigger as they go, they get more considered.
