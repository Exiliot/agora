## Notable decisions

- **Idempotent send.** `POST /api/friend-requests` reads-before-inserts and falls back to a read-after-violation path if a parallel request slips through. Duplicate senderŌåÆrecipient pairs return the existing pending request as 200 so the UI stays quiet. A 409 only fires for real conflicts (already-friends, either-side-banned).
- **Pair ordering is a helper, not a convention.** `pairKey(u1, u2)` returns the `(a, b)` tuple where `a < b`. Callers use it for both the `friendships` row and the `dm_conversations` shape; a single helper avoids sprinkling `u1 < u2 ? ... : ...` across route code.
- **Accept is one transaction.** Delete the request and insert the ordered friendship in the same tx; `onConflictDoNothing` guards against a race where the friendship was created by a mirror-path. Broadcasts fire after commit on both `user:<senderId>` and `user:<recipientId>`.
- **Reject is silent per FR-FRND-4.** The request row is deleted; no `user_ban` or `friend_request_cancelled` event is published to the sender.
- **Cancel publishes `friend.request_cancelled`** to the recipient so any pending notification UI can clear. The shared zod schema already carries this event type.
- **Ban is a three-step transaction.** Delete friend-requests in either direction, drop the friendship row if any, insert the ban (idempotent via `onConflictDoNothing`). Broadcasts go to both user topics so each side's cache updates in-place without a refetch.
- **Search uses prefix `like lower(q || '%')`** with the `NOT EXISTS` correlated subquery to drop users who have banned the caller, caps at 25, and always excludes the caller themselves and soft-deleted users.
- **User profile is minimal** per the requirements: `{ id, username, createdAt }`. Presence, bio and friend-count live outside this surface.

## What's deliberately not here

- DM auto-creation on friendship acceptance. The messages module owns `dm_conversations`; this feature only touches the friendship state machine.
- Personal-dialog read-only enforcement on ban ŌĆō already handled by `canSendDm` in the messages module via `eitherSideBanned` / friendship lookup; no changes needed here.
- WS dispatcher integration ŌĆō all broadcasts go through the existing in-process bus; the connection manager already forwards `user:<id>` events.
- Frontend surface ŌĆō web agent's territory.
- DB-backed integration tests ŌĆō shared fixture still pending (same reason as rooms).

## Verification

- Logic tests written to match the rooms spec style (pure helpers + event publishers via a real `bus.subscribe`).
- Typecheck deferred; the worktree lacks `node_modules` and running it requires the symlink trick from the brief, which would dirty the tree before commit.

## Takeaways

- The `pairKey` / `areFriends` / `eitherSideBanned` triad turns out to be the whole ban-semantics vocabulary; wrapping them as one helper module keeps the route code declarative.
- Machine error codes (`code: 'already_friends'`, `code: 'banned'`, `code: 'not_recipient'`, ŌĆ”) pair naturally with HTTP status; the UI branches on `code` and shows the human message.
- Idempotent-send-by-reread is simpler than trying to catch every race at the SQL layer and uses one fewer error path.
