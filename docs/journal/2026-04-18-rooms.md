## Notable decisions

- **Name uniqueness via DB unique index, not a pre-check.** The schema already has a case-insensitive unique index on `rooms.name`. Pre-checking would open a TOCTOU race; catching the duplicate-key error and returning 409 is both correct and one fewer round-trip on the happy path.
- **Ban list uses drizzle's `alias`.** First draft did a raw SQL template for the two joins against `users`; refactored to `alias(users, 'target')` + `alias(users, 'banner')` so the result type is inferred, no casts.
- **Remove-member = ban is one transaction.** Insert the ban row (idempotent via `onConflictDoNothing`) then delete the membership. Broadcasts happen after the commit so consumers never see a half-applied state.
- **Owner-cannot-leave returns 409 with `code: 'owner_cannot_leave'`.** Machine code separate from the human-readable message so the UI can branch on it.
- **Room detail carries `myRole`.** The shared `roomDetail` schema doesn't model this field, but the caller needs it to pick which moderation affordances to render. Kept as an additional response field rather than mutating `@agora/shared` (out of scope for this agent).
- **Private-room detail is 403 for non-members** (before ban check would even fire). Public rooms are readable by any signed-in user so the catalogue can link into them.
- **Delete room relies on FK cascades.** `ON DELETE CASCADE` from `rooms` down through memberships, bans, invitations, and messages means the transaction is a single `DELETE FROM rooms WHERE id = $1`. Actual file bytes on disk are another agent's territory.

## Takeaways

- Drizzle's `alias` is the right tool for same-table self-joins; reach for it before raw SQL.
- Keeping permission helpers parametric over `db | tx` means routes that open a transaction don't need a second code path for the same question.
- Machine codes in error responses (`code: 'owner_cannot_leave'` etc.) take nothing to add and save the frontend a regex race.
