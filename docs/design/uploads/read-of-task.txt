My read of the task

  A classic web chat à la mid-2000s IRC/rocket.chat: rooms, DMs, presence, friends, file sharing, moderation. Hard, load-bearing constraints:

  ┌──────────┬─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │   Area   │                                                     Non-obvious demand                                                      │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Realtime │ WebSockets + server-side in-memory queue (Denis repeated this twice). Message delivery <3s, presence <2s.                   │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Sessions │ Server-side, DB-backed, per-browser, individually revocable. JWT explicitly discouraged. Persistent across browser restart. │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Presence │ Per-user, aggregated across tabs. AFK after 1min idle on all tabs. Offline only when all tabs close.                        │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Scale    │ 300 concurrent users, rooms up to 1000, 10k+ message history per room, infinite scroll.                                     │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Files    │ Local FS (mounted volume), 20MB file / 3MB image caps, ACL tied to current room membership – lose access → lose files.      │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Bans     │ Two levels: user-to-user (personal DMs freeze, history remains read-only) and room-level (treated as ban, history hidden).  │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Delivery │ Public GitHub, main branch, git clone && docker compose up. ~50% of past submissions failed this – don't be a statistic.    │
  ├──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Optional │ Jabber/XMPP federation with two-server docker-compose + load test. Save for the end.                                        │
  └──────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

  Proposed tech stack

  My primary recommendation, followed by an alternative and what I'd rule out.

  Primary: Node 24 + TypeScript + Fastify + Postgres + React/Vite (pnpm monorepo)

  Alternative to consider: NestJS + Postgres if you prefer more structure. Tradeoff: more boilerplate, more patterns for the agent to keep consistent, but very uniform shape – every feature looks the
  same. I'd take Fastify for speed.

  What I'd rule out: Next.js (overkill for a pure SPA chat, App Router complicates WS), SQLite (single-file fine for dev, but Denis said Postgres is preferred), Redis (unnecessary for this scale and adds
  compose complexity), Socket.io (spec says WebSockets; Socket.io's fallbacks add weight), JWT-only auth (already covered).
